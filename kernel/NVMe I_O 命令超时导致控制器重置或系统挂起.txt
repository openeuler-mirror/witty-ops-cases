#标题
NVMe I/O 命令超时导致控制器重置或系统挂起

#内核版本
6.1（初始报告），涉及 6.7, 6.8, 6.11, 6.12.22, 6.14.5 等多个版本

#问题现象
- 系统日志中出现 "nvme nvme0: I/O ... timeout, aborting" 和 "reset controller" 信息
- 随后出现 "Device not ready; aborting reset, CSTS=0x1" 或 "Abort status: 0x371"
- 系统发生 I/O 阻塞、文件系统变为只读、系统冻结或 NVMe 设备从系统中消失
- 触发场景多样：高负载读写（如 rsync, btrfs scrub）、从挂起状态唤醒或无明显规律

#问题根因
该问题由多种因素独立或共同导致，具体取决于硬件型号与内核版本：
1. **固件缺陷**：部分 SSD（如 Samsung 990 Pro, Solidigm P44 Pro, SkHynix P41）固件在特定 I/O 模式或电源状态下无法在规定时间（默认 30 秒）内响应命令。
2. **内核调度回归**：内核 6.12.22 及之前版本存在 I/O 调度顺序问题，导致部分 SSD 固件预测算法失效引发超时，相关修复补丁为 `e70c301faece15b618e54b613b1fd6ece3dd05b4`。
3. **电源管理问题**：APST 或 ASPM 功能导致设备进入深度睡眠后无法正常唤醒。
4. **硬件链路问题**：个别案例中 PCIe 插槽布线或兼容性问题导致链路降速，引发传输超时。

#解决方案
1. **升级内核**：若使用 6.12.x 版本，请升级至 6.12.25 或更高版本以修复调度顺序问题；建议使用 6.13 及以上版本。
2. **调整内核启动参数**：添加 `nvme_core.default_ps_max_latency_us=0` 禁用 APST，或添加 `pcie_aspm=off` 禁用 PCIe 链路电源管理。
3. **更换 I/O 调度器**：尝试将 NVMe 设备的 I/O 调度器设置为 kyber 或 mq-deadline。
4. **硬件排查**：检查 SSD 固件是否为最新版本；尝试更换 PCIe 插槽或更换不同品牌/型号的 SSD。
5. **特定补丁**：针对特定受影响型号（如 Samsung 990 Pro），可能需要应用 `NVME_QUIRK_NO_DEEPEST_PS` 补丁（见 LKML 讨论）。
