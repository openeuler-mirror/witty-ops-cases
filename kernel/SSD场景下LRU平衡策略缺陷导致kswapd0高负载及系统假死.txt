#标题
SSD场景下LRU平衡策略缺陷导致kswapd0高负载及系统假死

#内核版本
4.0.4-303.fc22.x86_64（受影响版本包括 4.0.3 至 4.4.0-rc4）

#问题现象
- 系统在内存压力下（如运行 Eclipse、Firefox）出现严重卡顿甚至完全假死，需强制断电重启。
- kswapd0 和 kworker 进程占用 100% CPU（主要为 sys 时间）及大量 IO 资源。
- 触发 OOM Killer，日志显示 `bash invoked oom-killer`，`active_anon` 和 `inactive_anon` 占用大量内存，但 Swap 空间仍有大量剩余未被使用。
- 即使禁用 Swap，问题依然复现。

#问题根因
内核内存回收机制在平衡文件页和匿名页扫描压力时存在缺陷。在 SSD 场景下，IO 速度快，导致页面缓存快速刷新和重载。内存扫描器持续发现文件缓存是“可回收的”，因此优先回收文件页，导致严重的页面缓存抖动。扫描器未能及时转向扫描和换出匿名内存，导致系统在内存不足时长时间处于高负载回收状态而非触发 Swap 或 OOM，最终导致系统无响应。

#解决方案
暂无官方正式修复。开发者建议调整 LRU 平衡策略，按照 inactive file -> active file -> anon 的顺序进行回收，并引入基于抖动 IO 时间的 OOM 触发机制。评论中提及了一个尝试修复的补丁链接（http://lkml.iu.edu//hypermail/linux/kernel/1601.2/03564.html），但未确认是否合入主线。
