#标题
高并发TCP流量下内存回收路径触发spinlock死锁

#内核版本
3.10.7

#问题现象
- 系统在高并发、高流量 TCP 连接测试约 20 分钟后发生 spinlock 死锁，系统挂起无响应。
- 通过 NMI 获取的堆栈信息显示，多个 CPU 均阻塞在 `_raw_spin_lock` 函数，等待同一个锁地址（RDI: ffffffff80b936a0）。
- 典型调用栈为：`__do_page_fault` -> `handle_pte_fault` -> `__alloc_pages_nodemask` -> `try_to_free_pages` -> `shrink_slab` -> `prune_super` -> `put_super` / `grab_super_passive`。

#问题根因
系统在处理缺页异常时因内存不足触发直接内存回收，调用链进入 `shrink_slab` -> `prune_super` 试图收缩超级块对象时，尝试获取 `sb_lock` 自旋锁失败导致死锁。报告提交者推测可能是在文件 IO 路径已持有该锁的情况下，缺页异常处理流程再次尝试获取该锁导致死锁，但在报告中未给出最终确认的根因定论。

#解决方案
临时规避措施：调大 `sysctl vm.min_free_kbytes` 参数值，减少直接内存回收触发的概率。暂无正式修复补丁。
