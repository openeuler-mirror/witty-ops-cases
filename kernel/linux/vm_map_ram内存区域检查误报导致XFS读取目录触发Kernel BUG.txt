#标题
vm_map_ram内存区域检查误报导致XFS读取目录触发Kernel BUG

#内核版本
5.19-rc0

#问题现象
- 系统日志报错："usercopy: Kernel memory exposure attempt detected from vmalloc 'no area' (offset 0, size 1)!"
- 触发 kernel BUG at mm/usercopy.c:101!，导致系统崩溃或相关进程终止。
- 故障现场堆栈显示调用链为：usercopy_abort -> check_heap_object -> __check_object_size -> filldir64 -> xfs_dir2_leaf_getdents (或 xfs_dir2_block_getdents)。
- 在 s390x 和 aarch64 架构上均可复现，通常在执行 xfstests (如 xfs/294) 或使用 rm/find 命令遍历 XFS 目录时触发。

#问题根因
XFS 驱动使用 `vm_map_ram()` 接口映射内存，该接口不会分配 `vm_struct` 结构体。内核在 `mm/usercopy.c` 的 `check_heap_object` 函数中检查 vmalloc 地址时，调用了 `find_vm_area(ptr)`。由于 `vm_map_ram` 映射的内存区域没有对应的 `vm_struct`，该函数返回 NULL，导致内核误判为“Kernel memory exposure attempt”（内核内存泄露风险），从而触发 `usercopy_abort` 引发 BUG。这是一个安全检查机制的误报。

#解决方案
修改 `mm/usercopy.c` 中的检查逻辑，使用 `find_vmap_area((unsigned long)ptr)` 替代 `find_vm_area(ptr)`，因为 `find_vmap_area` 能够正确识别 `vm_map_ram` 映射的区域。同时需要在 `mm/vmalloc.c` 中导出 `find_vmap_area` 符号。开发者 Matthew Wilcox 已提供修复补丁，测试验证该补丁可解决问题。
