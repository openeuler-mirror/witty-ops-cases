#标题
并发回收导致clear_inode中nrpages断言失败触发kernel BUG

#内核版本
2.6.32-358.el6.x86_64

#问题现象
- 系统触发 kernel BUG，日志显示 "kernel BUG at fs/inode.c:329!" 及 "invalid opcode: 0000 [#1]"。
- 故障发生在 kswapd1 进程上下文中。
- 调用栈显示路径为：clear_inode -> generic_delete_inode -> iput -> dentry_iput -> d_kill -> __shrink_dcache_sb -> shrink_dcache_memory -> shrink_slab -> balance_pgdat -> kswapd。
- 同时 kswapd0 进程卡在 _spin_lock_irq，正在操作同一个 inode。

#问题根因
kswapd0 和 kswapd1 并发操作同一个 inode 引发竞态条件。kswapd0 在 __remove_mapping 中持有 mapping->tree_lock 进行页面回收；而 kswapd1 执行 truncate_complete_page 时，在检查 page->mapping 是否匹配的过程中未持有 tree_lock 锁。这导致 clear_inode 函数执行 BUG_ON(inode->i_data.nrpages) 检查时，页面可能尚未从 page cache 中正确移除或计数器未更新，nrpages 值不为 0，从而触发断言失败导致系统 panic。

#解决方案
报告者提出在 mm/truncate.c 的 truncate_complete_page 函数中增加 spin_lock_irq(&mapping->tree_lock) 锁保护以修复竞态。截止报告时间，Bug 状态为 NEW，尚未合并修复补丁。
