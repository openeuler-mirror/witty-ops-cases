#标题
UBIFS恢复过程处理ECC错误不当导致TNC数据不匹配

#内核版本
5.6.0-rc2

#问题现象
- 在包含 ECC 错误的 Flash 上挂载 UBIFS 成功后，执行 `du -h` 操作触发内核报错。
- 内核日志显示：`UBIFS error (ubi0:0 pid 1719): ubifs_read_node [ubifs]: bad node type (255 but expected 2)`。
- 读取到的节点数据全为 `0xff`，Call Trace 指向 `ubifs_read_node` -> `ubifs_tnc_read_node` -> `tnc_read_hashed_node` -> `ubifs_readdir`。

#问题根因
在特定场景下（GC LEB 42 为日志中最后一个 GC bud 节点，且以 inode 节点结尾），若该 LEB 中的 inode 节点发生 ECC 错误（CRC 损坏），UBIFS 的恢复逻辑处理不当。恢复过程丢弃了损坏的节点，但未能正确维护 TNC（Tree Node Cache）的一致性，导致后续读取时 TNC 指向了错误或无效的数据区域，引发节点类型不匹配错误。

#解决方案
应用修复补丁后，挂载过程中扫描 LEB 时会检测到 ECC 错误并打印 `ubifs_scanned_corruption` 日志，随后挂载失败并返回 "Structure needs cleaning" 错误，从而避免文件系统进入不一致状态。
