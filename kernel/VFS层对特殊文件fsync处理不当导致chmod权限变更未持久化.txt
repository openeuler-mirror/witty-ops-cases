#标题
VFS层对特殊文件fsync处理不当导致chmod权限变更未持久化

#内核版本
4.18 ~ 4.20.0+

#问题现象
- 对 FIFO 等特殊文件执行 chmod 修改权限后，调用 fsync 并模拟系统崩溃。
- 重启重新挂载后，发现文件权限未变更，回退到修改前的值（如预期 0400，实际仍为 0644）。
- 该问题影响所有特殊文件（块设备、字符设备、Unix domain socket、FIFO）及所有 Linux 文件系统。

#问题根因
VFS 层为特殊文件提供了 `file_operations` 结构体，但底层文件系统（如 ext4）未收到关于这些文件的 `fsync()` 调用通知。由于特殊文件的 `file_operations` 中未定义 `fsync` 函数，导致对特殊文件调用 `fsync()` 实际为空操作，未将元数据（如权限位）刷新到磁盘。这是 Linux 内核长期存在的行为，与 `fsync` 手册页描述存在偏差。

#解决方案
暂无代码修复。开发者认为修改内核实现成本较高且缺乏实际用户场景支持，倾向于修改 man page 以记录现有实现行为，或将其视为低优先级的 API 语义纯度问题。
