#标题
urma: 拓扑更新逻辑缺陷导致数据不一致或更新失败

#内核版本
OLK-6.6

#问题现象
在更新 URMA（User Mode Remote Access）聚合拓扑信息时，若新旧拓扑中存在相同 ID 的节点但设备或链路属性不一致（非新增场景），内核未进行一致性校验，可能导致拓扑数据被错误覆盖或处于不一致状态。同时，在 `ubcore_cmd_set_topo` 流程中，若创建 jetty 资源失败，错误处理路径会错误地删除全局拓扑表，导致已有拓扑信息丢失。

#主要错误日志
- "dev %d is not the same\n"
- "link is not the same, new: peer_node[%u]/peer_iodie[%u]/peer_port[%u], old: peer_node[%u]/peer_iodie[%u]/peer_port[%u]\n"
- "update link info fail."
- "update dev info fail."
- "update link info failed\n"
- "update dev info failed\n"
- "Failed to create jetty rsrc\n"

#问题根因
1. **逻辑校验缺失**：`update_dev_info` 和 `update_link_info` 函数在处理更新时，仅处理了“旧无效新有效”的新增场景，未处理“新旧均有效”的场景。当新旧拓扑中同一位置的设备或链路均有效但属性不同时，缺少校验逻辑，直接跳过处理或进行不合理的拷贝，导致拓扑信息不一致。
2. **更新流程遗漏**：`update_dev_info` 原返回值为 void，导致上层调用者 `ubagg_update_topo_info` 和 `ubcore_update_topo_map` 无法感知更新失败，无法及时中止错误的拓扑更新操作。
3. **错误处理冗余**：`ubcore_cmd_set_topo` 函数在 `ubcore_create_jetty_rsrc` 失败时，错误地调用了 `ubcore_delete_global_topo_map()`，误删了全局拓扑信息。

#解决方案
1. **增强拓扑更新校验**：修改 `update_dev_info` 和 `update_link_info` 函数，增加对“新旧均有效”场景的判断。若新旧设备或链路均有效但属性不一致，返回错误（-1）；若一致则不做处理。
2. **完善返回值检查**：将 `update_dev_info` 返回类型改为 int，并在上层调用逻辑中增加对其返回值的检查，若更新失败则返回 -EINVAL 并中止流程。
3. **修正错误处理路径**：移除 `ubcore_cmd_set_topo` 中 `ubcore_create_jetty_rsrc` 失败时的 `ubcore_delete_global_topo_map()` 调用，防止误删全局拓扑。
