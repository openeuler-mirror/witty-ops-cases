#标题
cifs: 解析相对于导出路径的原生符号链接错误

#内核版本
OLK-6.6

#问题现象
Linux SMB 客户端在读取 Windows 原生符号链接时，如果该符号链接设置了 `SYMLINK_FLAG_RELATIVE` 标志且目标路径以路径分隔符（如 `\`）开头（即相对于 SMB 导出根目录的相对路径），客户端会将其错误地解释为 Linux 文件系统根目录（`/`）下的绝对路径。
这导致符号链接指向了错误的目录位置（通常路径不存在或指向非预期文件），使得通过符号链接访问文件失败，无法正确解析相对于 SMB 共享顶级目录的 Windows 相对符号链接。

#主要错误日志
未知

#问题根因
SMB 协议中，相对符号链接可以相对于当前目录或相对于 SMB 导出（共享）的顶级路径，这取决于目标路径的首字符是否为分隔符。Linux（POSIX）仅支持相对于当前目录的符号链接。
代码原先在处理此类相对于导出根目录的符号链接时，直接保留了路径开头的分隔符，导致 Linux 将其解析为从系统根目录 `/` 开始的绝对路径，而不是相对于 SMB 挂载点的路径。由于 SMB 挂载点通常不是系统根目录，导致路径解析逻辑错误。

#解决方案
修改符号链接解析逻辑，新增 `smb2_parse_native_symlink` 函数用于统一处理原生 SMB 符号链接。
针对相对于导出根目录的链接（首字符为分隔符），通过计算符号链接自身路径中的路径分隔符数量来确定目录深度（N），并在目标路径前添加 N 个 `../` 前缀。此举将“相对于导出根”的路径转换为 Linux 支持的“相对于当前目录”的路径，从而确保路径指向正确的 SMB 共享内位置。
