#标题
fs_name() 函数存在无符号整数下溢风险

#内核版本
OLK-6.6

#问题现象
在调用 `fs_name()` 函数并传入特定的 `index` 参数时，由于该参数被定义为 `unsigned int`，在循环遍历文件系统链表进行自减操作 `index--` 时，若 `index` 初始值为 0 或在遍历过程中减至 0 后继续操作，会导致无符号整数下溢（变为极大的正整数）。这会导致循环判断条件 `index <= 0` 永远无法正确触发或逻辑异常，进而导致函数行为不符合预期，可能返回错误的文件系统名称或引发内核状态不一致。

#主要错误日志
未知

#问题根因
函数 `fs_name()` 的参数 `@index` 类型为 `unsigned int`。原代码逻辑中使用 `index--` 操作配合 `if (index <= 0)` 进行判断。由于无符号整数永远不会小于 0，当 `index` 为 0 时执行自减操作会发生下溢，变为 `UINT_MAX`，导致循环判断条件失效或逻辑错误，从而引发潜在的安全或功能问题。

#解决方案
修改 `fs_name()` 函数的循环逻辑，将判断条件从 `index <= 0` 改为 `index == 0`，并在满足条件时通过 `break` 语句立即退出循环。此举避免了无符号整数的下溢风险，并修正了比较逻辑，使其符合无符号整数的特性。同时优化了返回值处理逻辑，初始化 `res` 为 `-EINVAL`，仅在成功匹配 `index` 为 0 且获取模块引用时将其置为 0，确保正确返回错误码。
