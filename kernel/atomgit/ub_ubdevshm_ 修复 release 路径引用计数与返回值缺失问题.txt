#标题
ub/ubdevshm: 修复 release 路径引用计数与返回值缺失问题

#内核版本
OLK-6.6

#问题现象
1. 在 `ubdevshm_acquire_uba` 函数中，当访问上下文的获取引用计数（acquire_refcnt）为零时，虽然打印了错误日志并调用了 release 操作，但函数未返回错误码，导致上层调用者误认为操作成功，可能引发后续逻辑异常。
2. 在 `ubdevshm_release_uba` 流程中，特定代码路径下存在 shm_container 对象引用计数泄漏，导致对象无法被正确释放，造成内存泄漏。
3. `find_get_access_ctx_by_id` 相关逻辑中对句柄的处理存在潜在风险。
4. 模块描述信息不准确，且缺少模块作者信息。

#主要错误日志
- `pr_err("ctx acquire ref is zero\n");`
- `pr_err("find access ctx inner failed\n");`
- `pr_err("uba is NULL\n");`
- `pr_warn("refcnt is not zero, refcount is fail.\n");` (修复后将日志级别调整为 warn)

#问题根因
1. **返回值缺失**：在 `ubdevshm_acquire_uba` 函数的错误处理分支中，检测到 `acquire_refcnt` 为零后执行了资源释放操作，但遗漏了设置返回值 `ret = -ENOENT`，导致函数返回 0（成功）。
2. **引用计数泄漏**：`ubdevshm_release_uba` 函数逻辑混乱，在 `simple mode` 分支中未获取 shm_container 对象，却试图在通用退出路径 `out` 标签处释放它；同时 `grant mode` 的错误处理路径也存在重复释放或遗漏释放的不一致问题。这是由于代码逻辑结构未区分不同模式下 container 对象的获取状态导致的。
3. **其他问题**：代码缩进格式不规范、调试日志打印了多余的指针地址、模块元数据描述不准确。

#解决方案
1. 在 `ubdevshm_acquire_uba` 的错误分支补充 `ret = -ENOENT;`，确保错误状态正确向上传递。
2. 重构 `ubdevshm_release_uba` 的实现逻辑，提取公共代码到新函数 `find_get_shm_container_context` 中。该函数统一处理 simple mode 和 grant mode 下 container 和 access_ctx 的查找与引用获取，确保所有成功返回的路径都持有有效的 container 引用，且所有错误路径和退出路径都能正确、单一地调用 `shm_container_put` 释放引用，解决内存泄漏问题。
3. 修正 `ubdevshm_register_segment` 中不合理的引用释放逻辑。
4. 清理代码格式，修正 `MODULE_DESCRIPTION` 内容，补充 `MODULE_AUTHOR` 信息。
