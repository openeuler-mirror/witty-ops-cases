#标题
udma 模块 xarray 初始化标志缺失及 rx close 流程异常修复

#内核版本
OLK-6.6

#问题现象
1. 在驱动卸载或设备移除过程中，`udma_remove` 函数可能因 `udma_close_ue_rx` 执行失败而导致卸载流程卡死或报错回退。
2. 驱动内部使用的 xarray 表（如 `jfc_table`、`tpn_ue_idx_table` 等）在多线程并发访问场景下，因锁类型初始化不正确可能导致并发冲突或数据损坏。
3. 内存分配失败路径中，存在指针未置空导致潜在悬挂指针风险。
4. 调试模式下高频打印可能造成日志刷屏。

#主要错误日志
- "udma close ue rx failed in remove process."（推断，由 dev_err_ratelimited 调用推断）
- "pin kernel buf failed, ret = %d."
- "Failed to set active tp msg, ret %d."

#问题根因
1. **初始化标志缺失**：部分 xarray 表（如 `jfc_table`、`jetty_grp_table`、`npu_nb_table`、`tpn_ue_idx_table`）在并发场景下需要使用 IRQ 安全锁（`XA_FLAGS_LOCK_IRQ`），但代码中统一使用了无标志的 `xa_init`，导致锁类型不匹配，存在并发隐患。
2. **卸载流程不合理**：`udma_remove` 中调用 `udma_close_ue_rx` 失败时直接返回错误，未进行重试，导致设备无法正常卸载。
3. **资源清理不完整**：在 `udma_alloc_normal_buf`、`copy_out_cqe_data_from_user`、`udma_unpin_sw_db` 等函数的错误处理或释放路径中，内存释放后未将指针置为 NULL，存在悬挂指针风险。
4. **日志打印频率过高**：`udma_ctrlq_store_one_tpid` 等函数在高频调用路径下使用 `dev_info` 打印，可能导致内核日志缓冲区被刷屏。

#解决方案
1. **修正 xarray 初始化标志**：
   - 在 `udma_init_udma_table` 和 `udma_init_udma_table_mutex` 函数中新增 `irq_lock` 参数，根据调用者需求决定是否使用 `xa_init_flags(&table->xa, XA_FLAGS_LOCK_IRQ)` 初始化。
   - 在 `udma_init_tables` 中，为 `jfc_table`、`jetty_grp_table`、`npu_nb_table`、`tpn_ue_idx_table` 等表传递 `true` 参数以启用 IRQ 锁。
2. **优化卸载流程**：
   - 在 `udma_remove` 中引入重试机制，当 `udma_close_ue_rx` 失败时进行睡眠等待并重试，最大等待时间限制在 800ms，并使用 `dev_err_ratelimited` 打印日志，避免死循环。
3. **完善指针清理**：
   - 在内存释放路径（如 `kfree`、`vfree` 后）显式将指针置为 NULL（如 `buf->aligned_va`、`sq->wrid`、`db->page` 等），防止悬挂指针。
4. **日志限流**：
   - 将高频调试路径下的 `dev_info` 替换为 `dev_info_ratelimited`，抑制日志风暴。
5. **其他修复**：
   - `udma_mmap_hugepage` 中补充 `VM_WIPEONFORK` 标志。
   - 移除不必要的调试代码（如 `udma_dfx_ctx_print` 调用）。
