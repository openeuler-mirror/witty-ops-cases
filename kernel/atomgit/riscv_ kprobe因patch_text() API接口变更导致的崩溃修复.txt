#标题
riscv: kprobe因patch_text() API接口变更导致的崩溃修复

#内核版本
OLK-6.6

#问题现象
在 RISC-V 架构下使用 Kprobe 功能时，由于 `patch_text()` 函数接口定义发生变更（第三个参数由指令数量改为字节长度），而调用方（如 kprobes.c、livepatch.c）仍按旧接口传递指令数量（如传值为 1），导致内核写入错误的指令长度，引发 Kprobe 崩溃。

#主要错误日志
未知

#问题根因
`patch_text()` 函数的 API 接口进行了重构，其第三个参数由 `int ninsns`（指令数量）变更为 `size_t len`（字节长度）。然而，Kprobe、Livepatch 及 BPF JIT 等模块的调用代码未同步适配此变化，仍将指令数量（通常为 1）作为参数传递，导致内核误将极小的数值（如 1 字节）作为写入长度，破坏了正常的指令修补流程。

#解决方案
同步适配 `patch_text()` API 接口变更，修正所有调用处的参数传递：
1. 修改 `arch/riscv/kernel/probes/kprobes.c`，利用 `GET_INSN_LENGTH` 计算指令实际字节长度并传递给 `patch_text`。
2. 修改 `arch/riscv/kernel/livepatch.c`，将传递的参数从指令数量 `1` 修正为字节长度 `4`。
3. 修改 `arch/riscv/net/bpf_jit_comp64.c`，将传递的参数从指令数量宏修正为字节长度宏。
4. 重构 `arch/riscv/kernel/patch.c` 内部逻辑，直接使用传入的长度 `len` 进行写入，移除旧的指令计数循环逻辑。
