#标题
CPU 热插拔后因内存分配失败导致 Uncore sysfs 节点丢失

#内核版本
OLK-6.6

#问题现象
在系统启动后，若将某个 Package 内的所有 CPU 下线随后再上线，该 Package 对应的 Uncore frequency sysfs 接口可能丢失。即使重试 CPU 上线操作也无法恢复 sysfs 节点，导致无法通过 sysfs 对相应 Package 的 Uncore 频率进行控制或查询。

#主要错误日志
未知

#问题根因
在 CPU 热插拔上线流程（`uncore_event_cpu_online`）中，代码逻辑顺序存在缺陷：先将当前 CPU 加入 `uncore_cpu_mask` 掩码，随后才调用 `uncore_freq_add_entry` 创建 sysfs 节点。
如果 `uncore_freq_add_entry` 因为内存分配失败等原因返回错误，函数虽然返回了错误码，但 `uncore_cpu_mask` 已经被置位。当后续再次尝试上线 CPU 时，由于检测到 `uncore_cpu_mask` 中已存在该 Package 的 CPU（判定为 `target < nr_cpu_ids`），代码会直接返回，跳过 sysfs 节点的创建，导致 sysfs 节点永久丢失。

#解决方案
调整 `uncore_event_cpu_online` 函数中的执行顺序，确保先调用 `uncore_freq_add_entry` 创建 sysfs 节点并检查返回值。只有在节点创建成功后，才将 CPU 设置到 `uncore_cpu_mask` 掩码中；若创建失败则直接返回错误，不修改掩码状态。
