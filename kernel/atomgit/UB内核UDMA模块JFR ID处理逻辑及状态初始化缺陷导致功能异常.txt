#标题
UB内核UDMA模块JFR ID处理逻辑及状态初始化缺陷导致功能异常

#内核版本
OLK-6.6

#问题现象
1. 在创建或激活 JFR (Job File Region) 时，如果涉及用户态配置的 JFR ID，内核可能因 ID 分配逻辑冲突或检查错误导致操作失败，返回 -EINVAL。
2. JFR 及 JFC 的内部索引指针在初始化时未清零，可能导致新创建的对象使用了上一次遗留的脏数据，引发业务处理逻辑异常。

#主要错误日志
- "user set id %u error, min %u max %u.\n" (推断：在删除的代码逻辑中，当配置 ID 越界时触发)
- "failed to set jfr id, id is invalid or used. cfg_id=%u.\n" (推断：在删除的代码逻辑中，当 ID 分配冲突时触发)

#问题根因
1. **JFR ID 分配逻辑冲突**：原实现在 `udma_alloc_jfr` 中将 `rq.id` 初始化为 `UDMA_INIT_JFR_ID (-1)`，并在 `udma_k_set_jfr_param` 中调用 `udma_set_jfr_id` 尝试通过 IDA 机制分配/保留用户指定的 ID。该逻辑在 `udma_active_jfr` 阶段也会重复检查或尝试分配，导致逻辑复杂且易出错，特别是在处理用户指定 ID 时可能出现不必要的冲突或释放错误。
2. **状态变量未初始化**：`udma_init_jfc_param` 中未初始化 `jfc->ci`，`set_jfr_param` 中未初始化 `jfr->rq.pi` 和 `jfr->rq.ci`，导致后续业务逻辑使用未清零的脏数据。

#解决方案
1. **简化 JFR ID 处理逻辑**：
   - 移除 `UDMA_INIT_JFR_ID` 宏定义及相关复杂判断逻辑。
   - 删除 `udma_set_jfr_id` 函数，不再在参数设置阶段进行 IDA 预分配。
   - 直接在 `udma_k_set_jfr_param` 中将用户态传入的 ID 值赋给 `udma_jfr->rq.id`。
   - 将 ID 分配/验证逻辑统一收敛到 `udma_active_jfr` 流程中调用 `udma_alloc_jfr_id` 完成。
2. **补全对象初始化**：
   - 在 `udma_init_jfc_param` 中显式初始化 `jfc->ci = 0`。
   - 在 `set_jfr_param` 中显式初始化 `jfr->rq.pi = 0` 和 `jfr->rq.ci = 0`。
