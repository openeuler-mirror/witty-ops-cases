#标题
ub: cdma 基于流的注销过程中的时序问题

#内核版本
OLK-6.6

#问题现象
在进行基于流的注销或设备重置过程中，由于状态检查条件判断不完整（仅检查 `CDMA_SUSPEND` 而忽略了 `CDMA_INVALID`），以及销毁流程中注销顺序与时序处理不当，导致系统可能出现如下异常：
1. 在设备处于无效状态时仍尝试进行硬件操作（如 post mbox），导致操作失败或设备无响应。
2. 在注销或重置并发场景下，访问已释放的资源引发内核空指针崩溃或数据访问错误。
3. 驱动未能正确跳过无效上下文的硬件清理步骤，导致重置流程阻塞或报错。

#主要错误日志
- `pr_warn("cdma device is not prepared, eid = 0x%x.\n", ...);`
- `pr_info("ioctl cdev is invalid.\n");`
- `dev_warn(cdev->dev, "cdev is resetting.\n");`
- `pr_info("wait dev invalid event success.\n");`
- `dev_err(cdev->dev, "post mbox to destroy jfc failed, id: %u.\n", jfcn);`
- `dev_info(cdev->dev, "resetting Ignore jfc ctx, jfcn = %u\n", jfcn);`
- `dev_info(cdev->dev, "resetting Ignore jfs ctx, id = %u.\n", sq->id);`
- `dev_err(cdev->dev, "jfc delete failed, jfcn = %u.\n", jfcn);`
- `dev_err(cdev->dev, "jfs delete failed, id = %u.\n", jfs->id);`

#问题根因
该故障由驱动内部状态机管理与资源注销流程的时序缺陷引起：
1. **状态检查不完整**：原代码在多处关键路径（如 `dma_get_device_list`、`cdma_open` 等）仅判断 `status == CDMA_SUSPEND`，未覆盖设备处于 `CDMA_INVALID` 状态的情况。当设备正在重置或已失效时，错误的判断导致程序继续执行非法硬件操作。
2. **销毁流程逻辑错误**：在 `cdma_destroy_and_flush_jfc` 和 `cdma_modify_and_destroy_jfs` 函数中，原逻辑通过 `!(ctx && ctx->invalid)` 判断来决定是否执行销毁，但这导致在某些需要强制清理或跳过硬件交互的场景下逻辑互斥。
3. **注销时序违规**：在设备卸载流程 `cdma_uninit_dev` 中，`ubase_reset_unregister` 被错误地放在资源释放之后执行，可能导致在资源已被释放的情况下仍收到重置相关的事件或回调，引发 UAF 或空指针访问。

#解决方案
1. **引入新状态并修正判断逻辑**：新增 `CDMA_INVALID` 枚举值，并将所有设备状态检查从 `status == CDMA_SUSPEND` 修改为 `status >= CDMA_SUSPEND`，确保在设备挂起或失效状态下拒绝新的操作请求。
2. **优化销毁流程的时序控制**：修改 `cdma_destroy_and_flush_jfc` 和 `cdma_modify_and_destroy_jfs` 逻辑，增加对 `CDMA_INVALID` 状态和 `ctx->invalid` 标志的显式检查。当检测到设备正在重置或上下文已失效时，直接跳过硬件销毁指令（如 post mbox），避免硬件操作错误，仅进行软件层面的资源释放。
3. **调整函数调用顺序**：在 `cdma_delete_ctp` 函数中增加 `invalid` 参数，确保在无效状态下不发送硬件命令。同时，在 `cdma_uninit_dev` 中将 `ubase_reset_unregister` 调用移至获取设备锁之前，确保注销流程开始时即取消重置回调注册，避免竞态条件。
