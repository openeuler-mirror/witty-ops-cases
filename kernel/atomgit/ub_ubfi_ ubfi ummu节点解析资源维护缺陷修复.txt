#标题
ub:ubfi: ubfi ummu节点解析资源维护缺陷修复

#内核版本
OLK-6.6

#问题现象
在解析 ubfi ummu 节点时存在资源维护问题，可能导致内核在处理 ACPI 设备或 fwnode 时出现引用计数泄漏或异常。具体表现为：
1. 在 `ummu_config_update` 配置更新失败时，未释放已获取的设备引用，导致设备引用计数泄漏。
2. 在 `acpi_update_ummu_config` 添加 fwnode 失败时，未清理已添加的前序资源，可能导致资源残留或后续初始化异常。
3. `__init` 限定符使用不当，导致相关函数在内核初始化完成后无法被正常调用或热插拔场景下失效。
4. 错误日志打印缺失关键参数，导致故障定位困难。

#主要错误日志
- `pr_err("can not get ubrt fwnode for index=%u, type=%d\n", index, type);`
- `dev_err(&pdev->dev, "failed to generate new device name\n");`
- `dev_err(&pdev->dev, "failed to rename device to %s: %d\n", new_name, ret);`
- `dev_err(dev, "update config failed, ret[%d]\n", ret);`
- `dev_err(dev, "update fwnode failed, ret[%d]\n", ret);`
- `pr_err("failed to add fwnode for type %d, ret[%d]\n", ummu_types[i].type, ret);`
- `pr_err("acpi get devices err, status[%u]\n", status);`

#问题根因
1. **资源引用计数泄漏**：在 `acpi_processor_ummu` 函数中，当 `ummu_config_update` 或 `ubrt_fwnode_set` 失败时，直接返回 `AE_CTRL_FALSE`，但未调用 `put_device(dev)` 释放之前获取的设备结构体引用，导致引用计数泄漏。
2. **错误处理路径不完整**：在 `acpi_update_ummu_config` 函数中，循环添加不同类型的 fwnode 时，如果中间某次添加失败或 `acpi_get_devices` 失败，代码缺少对已成功添加的前序 fwnode 的回滚清理逻辑。
3. **函数生命周期标记错误**：`ummu_set_proximity`、`ummu_add_resources` 等函数被标记为 `__init`，但在非初始化阶段或需要长期可用的场景下被调用，导致内核无法正确执行函数逻辑。

#解决方案
1. **修复引用计数泄漏**：在 `acpi_processor_ummu` 函数的错误处理分支中添加 `put_device(dev)` 调用，确保在配置更新失败时正确释放设备引用。
2. **完善错误回滚机制**：在 `acpi_update_ummu_config` 函数中引入 `goto rollback` 逻辑，在添加 fwnode 失败或 ACPI 设备获取失败时，遍历并删除已添加的 fwnode，确保资源一致性。
3. **移除不当的限定符**：移除 `ummu_set_proximity` 和 `ummu_add_resources` 函数的 `__init` 限定符，使其在内核运行的整个生命周期内均可被调用。
4. **优化日志与代码结构**：增强错误日志输出，包含 index 和 type 等关键信息；重构代码以统一处理不同类型的 ummu 设备（如 UMMU 和 UMMU_PMU），提高代码可维护性。
