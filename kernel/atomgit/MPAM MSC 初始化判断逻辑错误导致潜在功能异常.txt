#标题
MPAM MSC 初始化判断逻辑错误导致潜在功能异常

#内核版本
OLK-6.6

#问题现象
MPAM（内存系统资源分区与监控）模块在初始化过程中，对 MSC（Memory System Component）的初始化判断逻辑存在缺陷。在某些 CPU 在线场景下，MSC 可能被错误地标记为“已跳过（skipped）”或“已探测（probed）”，但实际上并未完成初始化。这会导致后续在使能特性（如 `mpam_enable_init_class_features` 或 `mpam_enable_merge_features`）时，跳过本应初始化的 MSC 或处理未正确初始化的 MSC，可能导致 MPAM 功能失效、配置不一致或内核状态异常。

#主要错误日志
未知

#问题根因
1. **错误的跳过逻辑**：在 `mpam_discovery_cpu_online` 函数中，当某个 CPU 不在 MSC 的可访问性掩码（`accessibility`）中时，原代码错误地将 `msc->probed` 设为 true 并设置 `msc->skipped` 为 true。这导致即使该 MSC 还有其他在线且可访问的 CPU 未处理，也被提前标记为“已处理”，从而跳过了实际的初始化流程。
2. **错误的判断标志**：后续的特性初始化流程中，使用 `ris->msc->skipped` 作为判断条件，这与 MSC 是否真正完成探测初始化的语义不符。同时，在 `mpam_enable` 中判断 `all_devices_probed` 时，未结合 `cpu_online_mask` 检查是否还有潜在的在线 CPU 可以访问该 MSC，导致判断不准确。
3. **冗余状态字段**：引入了不必要的 `skipped` 布尔字段来跟踪状态，增加了状态管理的复杂性并导致了逻辑混乱。

#解决方案
1. **移除错误的跳过标记逻辑**：在 `mpam_discovery_cpu_online` 中，当 CPU 不可访问 MSC 时，直接 `continue` 跳过当前 CPU，不再错误地修改 `msc->probed` 或 `msc->skipped` 状态，确保 MSC 保留初始化机会以便后续可访问的 CPU 进行处理。
2. **统一探测判断标准**：将代码中对 MSC 状态的判断统一修正为检查 `msc->probed` 标志位。
   - 在 `mpam_enable_init_class_features` 中，将 `if (!ris->msc->skipped)` 修改为 `if (ris->msc->probed)`，确保只处理已成功探测的 MSC。
   - 在 `mpam_enable_merge_features` 中，将 `if (ris->msc->skipped)` 修改为 `if (!ris->msc->probed)`，跳过未探测的 MSC。
3. **完善使能检查逻辑**：在 `mpam_enable` 函数中，增加对 MSC 可访问性与在线 CPU 掩码的交集检查。如果 MSC 未探测（`!msc->probed`），则计算其可访问的在线 CPU 掩码，若掩码非空则判定 `all_devices_probed = false`，确保所有可被在线 CPU 访问的 MSC 均完成探测。
4. **清理冗余字段**：移除 `struct mpam_msc` 中的 `skipped` 字段，简化状态管理。
