#标题
sunrpc 内核 TCP Socket 导致的 UAF（Use-After-Free）故障

#内核版本
OLK-6.6

#问题现象
在删除网络命名空间时触发 KASAN 检测到的 slab-use-after-free 错误，导致内核调用栈打印并可能引发系统不稳定。具体表现为在网络命名空间被销毁后，TCP 定时器处理函数仍然试图访问已释放的 `net` 结构体。

#主要错误日志
```
BUG: KASAN: slab-use-after-free in tcp_write_timer_handler+0x156/0x3e0
Read of size 1 at addr ffff888111f322cd by task swapper/0/0
CPU: 0 UID: 0 PID: 0 Comm: swapper/0 Not tainted 6.12.0-rc4-dirty #7
...
Call Trace:
 <IRQ>
 kasan_report+0xbd/0xf0
 tcp_write_timer_handler+0x156/0x3e0
 tcp_write_timer+0x66/0x170
 ...
 </IRQ>
```

#问题根因
sunrpc 模块创建的内核 TCP Socket 未持有网络命名空间的引用计数。当网络命名空间被删除时，其关联的 `net` 结构体被释放，但此时如果 TCP Socket 仍有待处理的定时器回调（例如因 FIN 包被丢弃导致的重传定时器），定时器处理函数会尝试访问已被释放的 `net` 结构体，从而触发 UAF。

#解决方案
在 sunrpc 的 Socket 创建路径中，针对 TCP 协议手动增加网络命名空间的引用计数。具体修改如下：
1. 在 `net/sunrpc/svcsock.c` 的 `svc_create_socket` 函数中，对于 TCP Socket，设置 `sk_net_refcnt` 为 1 并调用 `get_net_track` 获取命名空间引用。
2. 在 `net/sunrpc/xprtsock.c` 的 `xs_create_sock` 函数中，执行同样的引用计数持有操作。
这确保了在网络命名空间销毁期间，如果 TCP 定时器仍在运行，`net` 结构体不会被提前释放。
