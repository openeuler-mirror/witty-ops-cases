#标题
spi: sh-msiof: 修复全双工模式下最大 DMA 传输大小计算错误

#内核版本
OLK-6.6

#问题现象
在全双工 SPI 传输（同时包含发送和接收数据）场景下，使用 DMA 传输时，计算单次 DMA 请求的最大数据量存在逻辑错误。该问题导致最大传输限制被错误地覆盖为接收 FIFO 的限制值，而非发送与接收限制中的最小值。根据 Commit 描述，由于当前硬件配置中 tx_fifo_size 与 rx_fifo_size 通常相等，该缺陷在实际运行中未产生明显功能性影响，但逻辑上不正确。

#主要错误日志
未知

#问题根因
驱动程序在计算单次 DMA 请求的最大数据传输量时存在逻辑缺陷。当 SPI 传输同时包含发送（tx）和接收（rx）数据时，代码先根据 tx_fifo_size 计算限制，随后又被 rx_fifo_size 的计算值覆盖，导致最终只采用了接收 FIFO 的限制，而未取两者的最小值。正确的逻辑应当是取发送和接收 FIFO 限制中的最小值作为最大传输长度。

#解决方案
修改 `drivers/spi/spi-sh-msiof.c` 中的 DMA 传输长度计算逻辑：
1. 引入新变量 `max_wdlen` 并初始化默认值为 256。
2. 在进入 DMA 传输循环前，分别检查发送和接收缓冲区，将 `max_wdlen` 更新为当前值与对应 FIFO 大小的最小值（即 `min(max_wdlen, p->tx_fifo_size)` 和 `min(max_wdlen, p->rx_fifo_size)`）。
3. 在循环内部计算单次传输长度 `l` 时，直接使用预先计算好的 `max_wdlen`，确保在全双工模式下使用的是两个方向 FIFO 大小的最小限制，同时优化了代码结构，避免在循环中重复计算。
