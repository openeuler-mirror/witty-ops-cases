#标题
x86/topology: ACPI MADT 表 CPU 枚举顺序错误导致内核误判 BSP 并禁用 CPU

#内核版本
OLK-6.6

#问题现象
在 Dell XPS 16 9640 机型上，由于 BIOS 未遵循 ACPI 规范，MADT 表中的 CPU 枚举顺序错误。内核因此错误地将正常运行环境判断为 crash kernel 环境，导致禁用真实的引导处理器（BSP），造成系统功能异常或不可预测的行为。

#主要错误日志
- `pr_warn("Boot CPU APIC ID not the first enumerated APIC ID: %x != %x\n", ...)`
- `pr_warn("Enumerated BSP APIC %x is not marked in APICBASE MSR\n", apic_id);`
- `pr_warn("Assuming crash kernel. Limiting to one CPU to prevent machine INIT\n");`
- `pr_warn("Crash kernel detected. Disabling real BSP to prevent machine INIT\n");`
- `pr_warn(FW_BUG "APIC enumeration order not specification compliant\n");`

#问题根因
ACPI 规范要求平台固件应将引导处理器（BSP）列为 MADT 表中的第一个处理器条目，内核依赖此顺序在 crash kernel 中检测真实的 BSP 以避免发送 INIT IPI 导致机器复位。
但在该故障场景中，BIOS 忽略了此规则，以错误的顺序枚举 CPU。内核原有逻辑仅依赖 MADT 表的枚举顺序进行判断，未能正确识别当前引导 CPU 是否为真正的 BSP，从而错误地触发了针对 crash kernel 的处理逻辑，导致 CPU 被错误禁用。

#解决方案
在 `check_for_real_bsp` 函数中增加对 `IA32_APICBASE` MSR 寄存器中 BSP 位的校验，以修正判断逻辑：
1. 如果引导 CPU 的 APICBASE MSR 中 BSP 位置位，说明其确为 BSP，此时忽略 MADT 的错误枚举顺序，直接采纳该 CPU 为 BSP，并打印固件 Bug 警告。
2. 如果引导 CPU 被枚举为第一个但 BSP 位未置位，为安全起见（防止向真实 BSP 发送 INIT 导致复位），假定当前为 crash kernel 并将 CPU 数量限制为 1。
3. 如果引导 CPU 不在首位且 BSP 位也未置位，则维持原有 crash kernel 判断逻辑。
