#标题
net/sched: cls_fw 在共享块场景下变更过滤器时引发空指针崩溃

#内核版本
OLK-6.6

#问题现象
在配置了共享 block 的网卡上使用 cls_fw 分类器，且数据包标记命中特定规则时，内核触发 WARN_ON 并随后发生空指针崩溃。具体表现为：
1. 执行流量控制规则添加与删除操作过程中，内核日志报错 `WARNING: ./include/net/pkt_cls.h:88 at fw_classify`。
2. 紧接着触发内核空指针引用错误 `BUG: kernel NULL pointer dereference, address: 0000000000000038`。

#主要错误日志
```log
WARNING: ./include/net/pkt_cls.h:88 at fw_classify+0x244/0x250 [cls_fw], CPU#18: mausezahn/1399
...
BUG: kernel NULL pointer dereference, address: 0000000000000038
...
RIP: 0010:fw_classify+0x1fe/0x250 [cls_fw]
```

#问题根因
自引入 commit ed76f5edccc9 后，TC 过滤器在被加入共享块并发布到数据路径时，是在调用其 `->change()` 函数之前进行的。
对于 cls_fw 模块，这意味着在共享块场景下，一个通过“旧方法”创建的无效过滤器可能会在短暂的生命周期内被数据路径访问。此时该过滤器尚未通过验证逻辑销毁，导致 `fw_classify()` 函数错误地进入了旧式分类逻辑路径。
在该路径中，代码尝试获取 `q` (Qdisc) 并解引用 `q->handle`，但由于是在共享块上，`tcf_block_q(tp->chain->block)` 返回了 NULL，从而触发了空指针解引用。

#解决方案
在 `fw_classify()` 函数的“旧方法”分类逻辑中增加检查。
如果在代码尝试解引用 Qdisc 指针之前检测到当前 block 是共享 block (`tcf_block_shared(tp->chain->block)`)，则直接跳过旧式分类逻辑（返回 -1），避免对 NULL 指针的解引用操作，从而修复该故障。
