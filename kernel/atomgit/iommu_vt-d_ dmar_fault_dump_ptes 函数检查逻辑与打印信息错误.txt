#标题
iommu/vt-d: dmar_fault_dump_ptes 函数检查逻辑与打印信息错误

#内核版本
OLK-6.6

#问题现象
在发生 DMA 故障进行 DMAR 转储结构（translation structure）dump 时，dmesg 输出的日志信息存在歧义或误导，例如将“pasid table is not present”误报为“pasid table entry is not present”。
此外，当页表不存在时，代码逻辑未正确校验即调用 pgtable_walk，导致可能输出冗余且无意义的页表项内容（如全 0 的 pasid table entry），最终报错 "PTE not present at level 2"。修复后，系统能准确判断并直接提示 "scalable mode page table is not present"。

#主要错误日志
修复前的典型错误日志输出：
[  442.240357] DMAR: pasid dir entry: 0x000000012c83e001
[  442.246661] DMAR: pasid table entry[0]: 0x0000000000000000
...
[  442.300742] DMAR: PTE not present at level 2

修复后预期的正确日志：
[  357.254824] DMAR: scalable mode page table is not present

#问题根因
函数 dmar_fault_dump_ptes 存在三个逻辑缺陷：
1. 错误地使用 phys_to_virt() 的返回值来判断页表项是否存在，这是不正确的判断依据。
2. 日志打印描述混淆，使用了 "entry" 字样（如 "pasid table entry is not present"），导致无法区分是“表不存在”还是“表项不存在”。
3. 在调用 pgtable_walk() 进行页表遍历前，未检查页表本身是否存在，导致在页表缺失时仍尝试遍历并输出无效数据。

#解决方案
修改 drivers/iommu/intel/iommu.c 中的 dmar_fault_dump_ptes 函数：
1. 修正判断逻辑，通过检查页表项的 present bit 来判断有效性，而非依赖 phys_to_virt 返回值。
2. 更正日志打印字符串，去除 "entry" 字样，明确表示“表”不存在。
3. 在调用 pgtable_walk() 之前增加对 legacy mode 及 scalable mode 页表是否存在的检查，若不存在则提前返回并打印正确日志。
