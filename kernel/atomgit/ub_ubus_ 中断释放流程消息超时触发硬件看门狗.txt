#标题
ub:ubus: 中断释放流程消息超时触发硬件看门狗

#内核版本
OLK-6.6

#问题现象
在中断释放过程中，内核持有 `spin_lock_irqsave` 锁。此时若发送配置消息发生超时，由于锁未被释放且流程未返回，将导致系统触发硬件看门狗复位。

#主要错误日志
- `ub int type1 clear UB_INT_TYPE1_ENABLE failed`
- `ub int type1 clear UB_INT_TYPE1_EN_INT_NUM failed`
- `ub int type1 clear UB_INT_TYPE1_INT_DATA failed`
- `ub int type1 clear UB_INT_TYPE1_INT_ADDR_L failed`
- `ub int type1 clear UB_INT_TYPE1_INT_ADDR_H failed`

#问题根因
在中断释放流程中，内核持有自旋锁（`spin_lock_irqsave`）。当调用配置写入函数（如 `ub_cfg_write_byte` 等）发送消息时，如果发生超时（返回 `-ETIMEDOUT`），原有代码未对返回值进行检查和处理，导致流程继续执行或死锁，最终引发硬件看门狗触发。

#解决方案
在中断释放的清理函数 `ub_int_type1_clear_msg` 中，增加对配置写入函数返回值的检查。若返回值为 `-ETIMEDOUT`，则打印错误日志并立即返回，避免在持有锁的情况下继续执行导致看门狗超时。
