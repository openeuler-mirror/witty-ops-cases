#标题
iommu/vt-d: 调用 qi_submit_sync 提交 0 count 导致内核软死锁

#内核版本
OLK-6.6

#问题现象
当并发场景下，某线程调用 `qi_submit_sync` 提交 0 个无效化描述符（例如用于 DMA drain），同时另一线程提交非零描述符时，提交 0 count 的线程可能无法检测到无效化完成的信号，导致线程无限循环等待，进而引发内核软死锁。

#主要错误日志
未知

#问题根因
根本原因是描述符状态标志 `QI_DONE` 被用于两个冲突的目的：
1. 标记描述符已完成，准备被回收；
2. 硬件标记等待描述符已完成。

在并发场景下，`reclaim_free_desc()` 函数会回收状态为 `QI_DONE` 的描述符并将其置为 `QI_FREE`。如果线程 T1 在回收过程中错误地将线程 T2 的等待描述符状态从 `QI_DONE` 改写为 `QI_FREE`，T2 将无法观察到硬件写入的 `QI_DONE` 状态，从而陷入死锁。当提交 0 个描述符时，队列中缺少作为屏障的无效化描述符（状态为 `QI_IN_USE`），使得回收代码容易错误地释放属于其他提交者的描述符。

#解决方案
修改描述符状态管理逻辑，实现状态分离。在 `qi_submit_sync` 完成处理后，将描述符状态直接置为 `QI_FREE` 而非 `QI_DONE`。同时修改 `reclaim_free_desc()` 函数，仅回收状态为 `QI_FREE` 的描述符，确保硬件完成信号（`QI_DONE`）不会被回收逻辑覆盖，从而保证提交线程能正确检测到完成状态。
