#标题
[UB/UNIC驱动增强异常CQE统计与修复设备初始化及命令队列竞态]

#内核版本
[OLK-6.6]

#问题现象
1. UNIC网卡驱动在发送队列（SQ）处理完成队列元素（CQE）时，若出现异常状态（status或sub_status非0），缺乏针对性的统计计数，导致`tx_timeout`或发送丢包时无法快速定位是硬件异常还是软件问题。
2. 在特定硬件版本（非A0/K0）上，驱动初始化完成队列上下文（JFC CTX）时，`ceqn`字段写入位置错误（写入了8bit区域而非9bit区域），可能导致中断路由配置错误或设备行为异常。
3. UBASE驱动在与对端设备（如MUE）通信时，若对端命令队列尚未就绪返回`ENXIO`错误，驱动未将其转换为可重试的错误码，导致在ELR或探测阶段加载失败。
4. 邮箱命令处理流程存在统计盲区，无法监控超时、序列号错误等关键事件，且存在潜在的未初始化引用风险。

#主要错误日志
- `ubase_warn: "the requested space(%d) exceeds the remaining space(%d), csq ci: %u.\n"`
- `ubase_warn: "peer cmdq is not ready.\n"`
- `ubase_err: "failed to send mbox over cmdq req, opc = 0x%x, seq = %u, ret = %d.\n"`
- `ubase_err: "failed to wait mbox over cmdq resp, opc = 0x%x, seq = %u, ret = %d.\n"`
- `ubase_err: "wait proxy resp timeout, seq = %u.\n"`
- `ubase_err_rl: "incomplete mailbox events exist, mbx stats: %llu, %llu, %llu, %llu, %llu, %llu, %llu, %llu\n"`
- `ubase_info: "tx timeout, queue index: %u, state: %lu\n ... abn_cqe_total_cnt: %llu\n"`

#问题根因
1. **统计缺失**：`unic_sq_stats`结构体缺少对异常CQE（`abn_cqe`）的详细统计，导致异常流量被归入正常统计或丢失，无法支撑故障定位。
2. **寄存器位域定义错误**：`unic_jfc_ctx`结构体中`ceqn`字段位宽定义单一（8bit），未根据硬件版本（hw_ver）区分8bit/9bit场景，导致在支持9bit ceqn的硬件上写入错误位置。
3. **错误码处理不当**：`ubase_get_cmd_result`函数对固件返回的`ENXIO`错误码未作特殊处理，无法区分“对端未就绪”与“真实错误”，导致过早失败而非重试。
4. **初始化顺序与竞态**：部分mailbox操作中`atomic_set(&mailbox.count, 0)`缺失；`ubase_send_cmd`中锁的释放与计数器减减操作位置不当，可能引起竞态条件。

#解决方案
1. **新增异常统计**：在`unic_sq_stats`中新增`abn_cqe_total_cnt`及二维数组`abn_cqe_cnt`，在`unic_reclaim_sq_space`中判断CQE状态并计数，通过debugfs和`ethtool -S`导出统计信息。
2. **适配硬件版本**：引入`unic_jfc_support_ceqn9`函数检查硬件版本，在`unic_jfc_ctx`中使用union定义`dw2_ceqn8`与`dw2_ceqn9`两种位域结构，根据hw_ver选择正确的字段写入。
3. **优化错误处理**：在`ubase_get_cmd_result`中识别`ENXIO`错误，若判定为对端命令队列未就绪（E2E消息场景），则将错误码转换为`ETIMEDOUT`以触发上层重试机制。
4. **修复并发与初始化**：在`ubase_config_ctx_buf_to_hw`等函数中补充`atomic_set(&mailbox.count, 0)`初始化；调整`ubase_send_cmd`中锁与计数器的操作顺序，确保原子性；添加`mbx_stats`统计项以增强可观测性。
