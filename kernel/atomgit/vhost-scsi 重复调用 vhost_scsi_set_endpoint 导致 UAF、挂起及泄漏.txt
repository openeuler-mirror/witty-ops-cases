#标题
vhost-scsi 重复调用 vhost_scsi_set_endpoint 导致 UAF、挂起及泄漏

#内核版本
OLK-6.6

#问题现象
当用户空间多次调用 `vhost_scsi_set_endpoint` 且中间未调用 `vhost_scsi_clear_endpoint` 时，触发以下多种故障现象：
1. **Use-After-Free（UAF）**：若第二次调用未找到任何 tpg，会导致内核释放 `vs->vs_tpg`，但 vq 后端指针仍指向该已释放内存。后续发送 SCSI 请求时引发 UAF，可能导致内核崩溃或数据破坏。
2. **目录删除挂起**：导致 LIO/Target 层的 tpg 引用计数降至 -1，用户空间执行 `rmdir` 删除 tpg 目录时挂起。
3. **资源泄漏**：若第二次调用使用不同的 target name，会导致旧 target name 对应的 tpg 无法被清理，造成资源泄漏且无法移除。

#主要错误日志
- `vhost-scsi endpoint already set for %s.`（推断：由补丁新增的 pr_err 打印，用于拦截重复设置）
- `pr_err` 或内核日志中可能出现与引用计数异常或内存访问错误相关的提示（具体取决于 UAF 触发场景）。

#问题根因
`vhost_scsi_set_endpoint` 函数未对多次调用做防护，导致逻辑缺陷：
1. **UAF**：代码逻辑中，若第二次调用未匹配到 tpg，会直接 `kfree(vs->vs_tpg)`，但此时 vq 的 backend 指针尚未置空，仍指向该已释放内存。
2. **挂起**：在重复设置时，错误处理流程会对已在数组中的 tpg 执行 `target_undepend_item`，导致引用计数多次减少（降至 -1），后续清理操作无法正常进行。
3. **泄漏**：重复调用时直接覆盖了 `vs->vs_vhost_wwpn`（target name），导致后续 `vhost_scsi_clear_endpoint` 无法匹配到旧 target name 对应的 tpg，无法执行 undepend 操作。

#解决方案
在 `vhost_scsi_set_endpoint` 函数开头增加检查，若 `vs->vs_tpg` 已设置，则直接返回 `-EEXIST` 拒绝重复配置。同时，在 `vhost_scsi_clear_endpoint` 中清理 endpoint 时，清零 `vs->vs_vhost_wwpn`。此举强制要求变更配置前必须先调用 clear 接口，从而规避上述 UAF、引用计数异常及资源泄漏问题。
