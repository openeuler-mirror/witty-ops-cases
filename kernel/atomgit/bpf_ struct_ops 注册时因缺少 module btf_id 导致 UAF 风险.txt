#标题
bpf: struct_ops 注册时因缺少 module btf_id 导致 UAF 风险

#内核版本
OLK-6.6

#问题现象
在编译配置为 `CONFIG_MODULES=n` 的内核环境中，使用 BPF struct_ops（特别是包含 `struct module *owner` 成员的类型，如 `tcp_congestion_ops`）时，存在 Use-After-Free (UAF) 安全风险。由于缺少 `struct module` 的 btf_id，`bpf_try_module_get()` 无法正确进行引用计数操作。在系统启动或模块加载初始化阶段，内核会检测到此配置不兼容并拒绝注册该 struct_ops，同时打印警告信息。

#主要错误日志
- `WARN: resolve_btfids: unresolved symbol module`
- `bpf_struct_ops '%s' needs module support. Is CONFIG_MODULES enabled?` （推断格式，基于 pr_warn 上下文）

#问题根因
在 `CONFIG_MODULES=n` 的配置下，内核 BTF 信息中缺少 `struct module` 的 btf_id（解析 BTF 时报错 `unresolved symbol module`）。对于包含 `struct module *owner` 成员的 struct_ops（如 `tcp_congestion_ops`），BPF 子系统依赖该 btf_id 来区分是常规内核模块扩展还是 BPF struct_ops 扩展，从而在 `bpf_try_module_get()` 中执行正确的引用计数逻辑。由于 btf_id 缺失，引用计数机制失效，从而导致 UAF 问题。

#解决方案
在 `bpf_struct_ops_desc_init` 初始化流程中增加校验逻辑：检查 struct_ops 是否包含 `struct module *` 类型的成员，同时检查 `struct module` 的 btf_id 是否存在。如果存在该成员但 btf_id 缺失（此时 btf 解析为 forward declaration），则拒绝注册并返回 `-EOPNOTSUPP` 错误，同时打印警告日志提示用户启用 `CONFIG_MODULES`。为此，补丁将 `btf_type_is_fwd` 函数移动到头文件中供校验逻辑使用，并新增 `is_module_member` 辅助函数进行类型检测。
