#标题
ath11k: debugfs_fw_stats_request 函数 CPU 占用过高问题修复

#内核版本
OLK-6.6

#问题现象
系统出现 CPU 空转（burning CPU）现象，在 `ath11k_debugfs_fw_stats_request()` 函数中陷入死循环或忙等待状态。
通过 perf 分析可见，CPU 在 `do_syscall_64` -> `__sys_sendmsg` -> `ieee80211_dump_station` -> `ath11k_mac_op_sta_statistics` -> `ath11k_debugfs_fw_stats_request` 调用链中占用率极高（报告显示占比约 94.55%），其中大部分时间消耗在 `_raw_spin_lock_bh`、`__local_bh_enable_ip` 和 `_raw_spin_unlock_bh` 等自旋锁操作上。

#主要错误日志
未知

#问题根因
在 `ath11k_debugfs_fw_stats_request()` 函数中，原实现使用一个布尔变量 `ar->fw_stats_done` 配合无限循环（循环内检查条件并自旋锁操作）来等待固件统计请求完成。
如果因某种原因 `ath11k_update_stats_event()` 未能设置 `ar->fw_stats_done` 为 true，该函数将持续在循环中忙等待，直到长达 3 秒的超时结束，导致 CPU 资源被无意义地消耗在自旋锁的获取与释放上。

#解决方案
将原有的布尔标记变量 `fw_stats_done` 替换为内核完成量（`struct completion`）机制。
具体修改包括：
1. 将结构体中的 `bool fw_stats_done` 替换为 `struct completion fw_stats_done`。
2. 在初始化函数 `ath11k_fw_stats_init` 中调用 `init_completion` 进行初始化。
3. 在请求函数 `ath11k_debugfs_fw_stats_request` 中，使用 `wait_for_completion_timeout` 替代原有的轮询循环逻辑，直接阻塞等待完成或超时（3秒），从而让出 CPU。
4. 在事件处理函数 `ath11k_update_stats_event` 及 `ath11k_debugfs_fw_stats_process` 中，将设置 `ar->fw_stats_done = true` 改为调用 `complete(&ar->fw_stats_done)` 以唤醒等待进程。
