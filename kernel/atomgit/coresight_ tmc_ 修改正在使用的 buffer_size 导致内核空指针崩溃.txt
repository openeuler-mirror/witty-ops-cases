#标题
coresight: tmc: 修改正在使用的 buffer_size 导致内核空指针崩溃

#内核版本
OLK-6.6

#问题现象
在启用 Coresight 追踪功能的过程中，如果已启用 sink 和部分 source，此时尝试修改 buffer_size，随后启用其他 source 并读取数据，将导致内核崩溃。具体表现为 NULL pointer dereference（空指针引用），触发内核 panic/oops。

#主要错误日志
Unable to handle kernel NULL pointer dereference at virtual address 0000000000000028
Call trace:
 __tmc_etr_disable_hw+0x108/0x140 [coresight_tmc]
 tmc_read_prepare_etr+0xc0/0xd0 [coresight_tmc]
 tmc_open+0x60/0xa0 [coresight_tmc]
 ...
 tmc_etr_get_sysfs_trace+0x10/0x80 [coresight_tmc]
 vfs_read+0xcc/0x310

#问题根因
当 Coresight TMC 设备处于使用状态（sink 已启用）时，仍然允许通过 sysfs 修改 buffer_size。这导致在后续启用第二个 source 时，`tmc_etr_get_sysfs_buffer()` 会根据新的大小重新分配缓冲区。在追踪结束时，`tmc_etr_sync_sysfs_buf()` 会重置 `drvdata->sysfs_buf`，导致后续读取数据时触发 NULL 指针引用崩溃。

#解决方案
在 `buffer_size_store` 函数中增加并发保护与状态检查。在修改 buffer_size 之前，持有自旋锁（`drvdata->spinlock`）并检查设备模式（`drvdata->mode`）。如果设备当前未处于禁用状态（`CS_MODE_DISABLED`），即设备正在使用中，则直接返回 `-EBUSY` 拒绝修改，从而避免缓冲区在使用中被重新分配。
