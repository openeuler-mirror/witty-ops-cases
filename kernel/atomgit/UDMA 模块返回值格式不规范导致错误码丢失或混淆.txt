#标题
UDMA 模块返回值格式不规范导致错误码丢失或混淆

#内核版本
OLK-6.6

#问题现象
在 UDMA 驱动的多个子模块（jfc, jfs, jfr, jetty）中，创建或分配资源的接口（如 udma_create_jfc, udma_create_jfs 等）在遇到错误（如内存分配失败、ID 分配失败、参数校验失败）时，错误地返回了 NULL 或丢弃了具体的错误码。这导致上层调用者无法通过返回值准确感知具体的错误原因（如内存不足、无空间、参数无效等），可能导致上层逻辑处理异常或调试困难。

#主要错误日志
- "failed to alloc id, ret = %d.\n"（推断上下文：ID 分配失败时打印）
- "active jetty detail failed, ret = %d.\n"
- "the jetty of the type %u cannot be imported in exp.\n"
- "failed to alloc id for stars JFC.\n"
- "ida alloc failed %d.\n"
- "alloc jfr id range (%u - %u) failed, ret = %d.\n"
- "verify jfr param failed.\n"
- "jfs not support RC transmode.\n"

#问题根因
驱动代码中存在多处返回值处理不当的问题：
1.  内存分配失败或参数校验失败时，函数直接返回 NULL，而不是使用 ERR_PTR 机制封装具体的错误码（如 -ENOMEM, -EINVAL），导致错误原因丢失。
2.  在处理 IDA（ID Allocator）分配失败时，部分代码直接返回了 -ENOSPC，但根据 UDMA 的返回值规范或调用链需求，应将其转换为 -ENOSR，导致错误码语义不一致。
3.  部分错误处理路径直接丢弃了底层函数返回的错误码，导致无法向上传递真实的失败原因。

#解决方案
修改 UDMA 驱动相关函数的返回值处理逻辑，使其符合内核标准的错误返回机制：
1.  将所有内存分配失败（kzalloc/kcalloc）的返回值从 `return NULL` 修改为 `return ERR_PTR(-ENOMEM)`。
2.  将参数校验失败等其他错误路径的返回值从 `return NULL` 修改为 `return ERR_PTR(ret)` 或 `ERR_PTR(-EINVAL)`，确保错误码正确向上传递。
3.  统一 ID 分配失败时的错误码转换逻辑，将 `ida_alloc_range` 等接口返回的 -ENOSPC 转换为 -ENOSR 返回（`return ret == -ENOSPC ? -ENOSR : ret`）。
4.  修复 udma_create_stars_jfc 调用处未捕获返回值的问题，确保错误能被正确检测并跳转到错误处理流程。
