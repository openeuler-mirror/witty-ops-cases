#标题
fuse: fix dax truncate/punch_hole fault path

#内核版本
OLK-6.6

#问题现象
在 FUSE 文件系统使用 DAX 功能时，执行 truncate 或 punch_hole 操作期间，系统未能正确解除文件整个范围的映射。这可能导致在文件大小缩减或打洞操作后，仍存在对已释放磁盘块的 DMA 或远程访问，存在数据一致性风险或潜在的内存访问故障。

#主要错误日志
未知

#问题根因
代码中注释声称调用 `fuse_dax_break_layouts` 时传入 `dmap_end == 0` 会导致解除整个文件的映射，但实际实现逻辑并非如此。传入 `dmap_end == 0` 时，函数不会取消 `dmap_start` 之前的映射，且 `dax_layout_busy_page_range()` 也不会扫描该范围以检查是否正在进行的 DMA 访问。这导致 truncate/punch_hole 路径下无法正确失效整个文件的 DAX 映射。

#解决方案
修改 `fuse_dax_break_layouts` 函数的调用参数，将 `dmap_end` 参数从 `0` 修改为 `-1`（在内核中表示 `ULONG_MAX` 或全范围），确保能正确通知底层 DAX 层扫描并失效整个文件范围的映射，从而保证在 truncate 或 punch_hole 操作期间文件布局的安全变更。同时删除了代码中关于 `dmap_end == 0` 的错误注释。
