#标题
x86/ia32: NULL 选择器值 0~3 未保持不变导致信息泄露

#内核版本
OLK-6.6

#问题现象
在非 FRED（Flexible Return and Event Delivery）系统的 x86 架构下，用户态程序可以通过加载非零的 NULL 选择器（值为 1、2 或 3），利用中断/异常返回时内核对段寄存器清零的行为，来探测内核是否发生了中断或异常。这种机制构成了一个信息泄露漏洞，允许用户态程序监控系统中断/异常的发生。此外，`reload_segments` 与 IRET 指令对选择器值的处理逻辑相互抵消，可能导致预期的段寄存器重载操作失效（表现为无操作 nop）。

#主要错误日志
未知

#问题根因
GDT 的第一个描述符保留为 NULL 描述符，段选择器的最低两位（RPL 位）不用于索引 GDT，因此选择器值 0~3 均指向 NULL 描述符，它们都是有效的 NULL 选择器值。
在之前的代码实现中：
1. `reload_segments()` 函数在加载段寄存器时，无条件地将选择器值的 RPL 位设置为 3（例如 `sc->gs | 0x03`），这导致原本的非零 NULL 选择器（如 1、2、3）被修改为 3。
2. 当发生中断或异常返回时，IRET 指令检测到段寄存器（ES、FS、GS、DS）的值为非零的 NULL 选择器（即值为 3），会将其清零。
上述两个行为结合导致：用户态加载非零 NULL 选择器后，若发生中断/异常，选择器值会被清零。用户态可通过检测选择器值是否变为 0 来推断是否发生了中断/异常，从而造成信息泄露。同时，`reload_segments` 设置 RPL 和 IRET 清零操作相互抵消，导致段重载实际上未生效。

#解决方案
修改 `arch/x86/kernel/signal_32.c` 中的逻辑，引入 `fixup_rpl()` 辅助函数。该函数判断如果选择器值小于等于 3（即 NULL 选择器），则保持原值不变；否则才设置 RPL 位为 3。此举确保 NULL 选择器值（0~3）在内核处理过程中保持不变，避免了 IRET 指令将其清零，从而关闭了该信息泄露通道，并修正了段重载的逻辑。
