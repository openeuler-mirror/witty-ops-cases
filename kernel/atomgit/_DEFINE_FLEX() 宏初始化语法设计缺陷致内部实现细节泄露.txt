#标题
_DEFINE_FLEX() 宏初始化语法设计缺陷致内部实现细节泄露

#内核版本
OLK-6.6

#问题现象
使用 `_DEFINE_FLEX()` 宏定义柔性数组结构体变量并试图静态初始化结构体成员时，必须在初始化列表中显式引用内部实现成员 `obj`（即 `.obj = { ... }`）。这种非标准的 C99 初始化方式泄露了宏的内部实现细节，导致代码可读性降低，调用复杂且不直观。

#主要错误日志
未知

#问题根因
`_DEFINE_FLEX()` 宏内部通过定义联合体来实现柔性数组的栈上分配，但在宏定义中直接将 `initializer` 参数拼接在联合体变量声明之后。这导致调用者无法直接对联合体内部的 `type obj` 成员进行初始化，而被迫在调用处显式指定 `.obj` 成员名。根本原因是宏定义未正确处理初始化列表的作用域，未将初始化参数自动绑定到联合体内部的 `obj` 成员上。

#解决方案
修改 `include/linux/overflow.h` 中的宏定义逻辑：
1. 调整 `_DEFINE_FLEX()` 宏的展开形式，将 `initializer` 参数封装在 `{ .obj initializer }` 中，使其自动指向联合体内部的 `obj` 成员。
2. 同步更新 `DEFINE_FLEX()` 宏的调用，移除显式的 `.obj` 前缀，改为直接使用标准的 C99 初始化风格（如 `= { .COUNTER = COUNT, }`）。
3. 明确 `initializer` 参数不再允许为空，必须传递至少 `= { }` 形式的参数。
