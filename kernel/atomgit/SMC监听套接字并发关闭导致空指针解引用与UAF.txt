#标题
SMC监听套接字并发关闭导致空指针解引用与UAF

#内核版本
OLK-6.6

#问题现象
在 SMC (Shared Memory Communications) 监听套接字关闭过程中，若并发处理 TCP 连接请求（如 SYN cookie 路径或正常握手路径），会导致内核崩溃。
具体表现为：
1. 空指针解引用：导致 Kernel Panic。
2. 释放后使用：访问已释放的 smc_sock 结构体字段（如 queued_smc_hs），导致系统崩溃或数据损坏。

#主要错误日志

#问题根因
`smc_tcp_syn_recv_sock()` 在 TCP 接收路径中被调用，它通过读取 `sk_user_data` 获取 `smc_sock` 指针。
根因在于 clcsock（TCP 监听套接字）与 smc_sock 是两个独立的对象，拥有独立的引用计数。当 SMC 监听套接字被并发关闭时，`smc_close_active()` 会在持有 `sk_callback_lock` 的情况下将 `clcsock->sk_user_data` 设置为 NULL，随后通过 `sock_put()` 释放 `smc_sock`。
此时 TCP 栈仍持有 clcsock 的引用使其存活，但这无法阻止 smc_sock 被释放。
竞态窗口如下：
- CPU A（软中断上下文）：正在处理 TCP 连接请求，尝试读取 `sk_user_data`。
- CPU B（进程上下文）：正在执行关闭操作，将 `sk_user_data` 置 NULL 并释放 smc_sock。
导致 CPU A 要么读到 NULL 引发空指针崩溃，要么读到悬空指针引发 UAF 崩溃。

#解决方案
采用 RCU 机制与引用计数保护来安全访问 `smc_sock`，避免重量级锁带来的性能问题：
1. 设置 `SOCK_RCU_FREE` 标志：在 SMC 监听套接字上设置此标志，确保 `smc_sock` 的内存释放延迟到 RCU 宽限期之后，保证在 `rcu_read_lock()` 临界区内访问时内存有效。
2. 使用 RCU 保护读取：将 `sk_user_data` 的读取修改为使用 `rcu_dereference_sk_user_data`（封装在新增的 `smc_clcsock_user_data_rcu` 函数中），并在 `rcu_read_lock()` 保护下进行。
3. 引用计数检查：读取到指针后，使用 `refcount_inc_not_zero(&smc->sk.sk_refcnt)` 尝试增加引用计数。如果引用计数已为零（关闭路径已完成），则安全退出；否则持有该对象直到处理完毕。
4. 相应地，在设置和清除 `sk_user_data` 的代码路径（如 `smc_listen`、`smc_close_active` 等）中，将直接赋值改为使用 `rcu_assign_sk_user_data` 系列接口。
