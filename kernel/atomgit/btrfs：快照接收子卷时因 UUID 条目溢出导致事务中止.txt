#标题
btrfs：快照接收子卷时因 UUID 条目溢出导致事务中止

#内核版本
OLK-6.6

#问题现象
对已接收的子卷进行大量快照创建操作时，触发 BTRFS_UUID_KEY_RECEIVED_SUBVOL 条目溢出，导致文件系统事务中止，文件系统被强制切换为只读模式，后续快照创建失败。用户空间报错 "Value too large for defined data type" 和 "Read-only file system"。

#主要错误日志
[251067.627338] BTRFS warning (device sdi): insert uuid item failed -75 (0x4628b21c4ac8d898, 0x2598bee2b1515c91) type 252!
[251067.630033] BTRFS: Transaction aborted (error -75)
[251067.630871] WARNING: fs/btrfs/transaction.c:1907 at create_pending_snapshot.cold+0x52/0x465 [btrfs]
[251067.687890] BTRFS: error (device sdi state A) in create_pending_snapshot:1907: errno=-75 unknown
[251067.689049] BTRFS info (device sdi state EA): forced readonly
[251067.689054] BTRFS warning (device sdi state EA): Skipping commit of aborted transaction.

#问题根因
当用户对先前已接收的子卷反复创建大量快照时，内核尝试向 BTRFS UUID 树中插入 BTRFS_UUID_KEY_RECEIVED_SUBVOL 条目。当条目数量达到叶节点存储上限时，btrfs_uuid_tree_add() 返回 -EOVERFLOW 错误。原有代码未处理该溢出错误，直接将其视为致命错误并触发了事务中止，导致文件系统变为只读模式。由于创建快照不需要 CAP_SYS_ADMIN 权限，普通用户即可利用此缺陷发起拒绝服务攻击。

#解决方案
在快照创建逻辑中，针对 btrfs_uuid_tree_add() 返回的 -EOVERFLOW 错误进行特殊处理，将其忽略并视为成功（ret = 0）。该处理方式是安全的，因为即使缺少该 UUID 条目，快照删除功能仍可正常工作（删除时忽略缺失条目），且 send/receive 操作也能通过回退查找 BTRFS_UUID_KEY_SUBVOL 键正常进行。此修复避免了非关键错误导致的文件系统只读切换。
