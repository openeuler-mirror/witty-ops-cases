#标题
net-timestamp: 使用 SOF_TIMESTAMPING_OPT_ID 时 sk_tskey 在错误路径下不可预测

#内核版本
OLK-6.6

#问题现象
当应用程序使用 SOF_TIMESTAMPING_OPT_ID 选项进行数据报时间戳标记时，如果在 sendmsg() 发送过程中发生错误（如内存分配失败、权限错误等），内核对 sk_tskey 的处理逻辑存在缺陷。这导致发送失败后，套接字的时间戳 ID 计数器（sk_tskey）与实际逻辑状态不一致，使得后续发送操作的时间戳 ID 出现跳跃或不连续，无法正确匹配时间戳与数据报，影响依赖时间戳 ID 的应用层逻辑判断。

#主要错误日志
未知

#问题根因
在修复前，sk_tskey 的自增操作（atomic_inc_return）发生在 __ip_append_data（IPv4/IPv6）函数的入口早期阶段。如果后续的数据处理流程（如内存分配、分片处理等）进入错误处理分支并返回错误，先前已经增加的 sk_tskey 计数器不会被回滚（decrement）。这导致即使数据包发送失败，时间戳 ID 也被消耗掉了，从而在错误路径下导致 sk_tskey 值变得不可预测。

#解决方案
将 sk_tskey 的自增操作移动到函数逻辑更靠后的位置（确保进入实质性数据处理阶段后再增加计数），并在函数的错误处理返回路径中增加对 sk_tskey 的回滚操作（atomic_dec）。具体修改如下：
1. 在 `net/ipv4/ip_output.c` 和 `net/ipv6/ip6_output.c` 中，删除原入口处的 sk_tskey 自增逻辑。
2. 将 sk_tskey 的自增操作移至 `cork->length += length;` 之后执行，并引入 `hold_tskey` 布尔变量记录是否执行了自增。
3. 在函数结尾的 `error` 标签或错误返回路径中，检查 `hold_tskey` 变量，如果为真则执行 `atomic_dec(&sk->sk_tskey)`，确保在发送失败时回滚计数器。
