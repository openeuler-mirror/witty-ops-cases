#标题
bnxt_en: 修复 HW GRO/LRO 导致的潜在数据损坏问题

#内核版本
OLK-6.6

#问题现象
使用 Broadcom bnxt_en 网卡驱动（57500 芯片）时，开启硬件 GRO/LRO 功能可能导致数据损坏。在故障发生时，可能会看到如下 TPA 聚合缓冲区不一致的错误日志：
`bnxt_en 0000:83:00.0 enp131s0np0: TPA end agg_buf 2 != expected agg_bufs 1`

#主要错误日志
- `bnxt_en 0000:83:00.0 enp131s0np0: TPA end agg_buf 2 != expected agg_bufs 1`
- `netdev_warn(bp->dev, "Unable to allocate agg ID for ring %d, agg 0x%x\n", ...)`

#问题根因
函数 `bnxt_alloc_agg_idx()` 在调用 `find_first_zero_bit()` 时，错误地传入了 `BNXT_AGG_IDX_BMAP_SIZE`（即 long word 的数量）作为位图大小参数，而该函数期望的是位数量（number of bits）。
这导致 `find_first_zero_bit()` 可能搜索越界或无法正确找到空闲位，进而返回并使用一个错误的、已被占用的 ID。当该错误 ID 与正在使用的 ID 冲突时，引发数据损坏。

#解决方案
1. 修正 `find_first_zero_bit()` 的第二个参数，将其从 long word 数量改为正确的位数量 `MAX_TPA_P5`。
2. 使用 `DECLARE_BITMAP()` 宏更规范地定义位图结构，替代手动计算数组长度的旧方式。
3. 增加防御性检查：如果无法找到空闲位（`idx >= MAX_TPA_P5`），返回 `INVALID_HW_RING_ID`，并在上层调用逻辑中检测到此无效 ID 时打印警告日志并调度 Ring Reset，避免继续处理导致数据损坏。
