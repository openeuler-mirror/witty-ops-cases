#标题
octeontx2-af: APR entry mapping error due to fixed LMT table size

#内核版本
OLK-6.6

#问题现象
当 PF（物理功能）和 VF（虚拟功能）的数量发生变化时，驱动程序使用固定大小映射 APR 表，导致 debugfs 中 LMTST map table 的显示条目不正确。如果实际配置超过了固定大小（128KB），可能导致内存映射越界或读取到错误的数据；如果实际配置较小，则造成资源浪费。

#主要错误日志
- "Failed to setup lmt map table mapping!!\n" (推断：当映射失败时可能输出)

#问题根因
当前实现使用固定的宏定义 `LMT_MAP_TABLE_SIZE` (128 * 1024) 来映射 APR 表。但实际上 APR 表的大小取决于 PF 和 VF 的数量组合。由于硬件配置中 PF 和 VF 的数量是可变的，固定大小的映射无法适应所有配置场景，导致映射区域与实际硬件配置不匹配。

#解决方案
移除固定的 `LMT_MAP_TABLE_SIZE` 宏定义，改为动态计算 APR 表大小。
1. 读取 `APR_AF_LMT_CFG` 寄存器获取配置值。
2. 从配置值中解析出 VF 和 PF 的数量（`vfs` 和 `pfs`）。
3. 根据公式 `pfs * vfs * LMT_MAPTBL_ENTRY_SIZE` 动态计算并映射正确大小的内存区域。
4. 同步更新 debugfs 中的遍历逻辑，使用解析出的 `apr_vfs` 变量替代原有的固定值，确保遍历索引计算正确。
