#标题
scsi: libsas: 修复因零地址PHY加入端口触发的内核BUG()及hisi_sas驱动多项稳定性问题

#内核版本
OLK-6.6

#问题现象
1. **内核崩溃**：在向新端口添加具有零地址（zero-address）的 PHY 时，触发 `BUG()` 导致内核 panic。
2. **控制器复位异常**：控制器复位后，未在用户侧禁用的 PHY 未被重新使能，导致链路无法恢复。
3. **PHY 状态异常**：PHY Ready 状态超时或设备移除时，PHY 未被重置，导致状态卡死。
4. **磁盘软复位异常**：ATA 磁盘软复位期间，未遵循规范在两次 H2D FIS 之间添加时间间隔，可能导致复位失败。
5. **设备启停失败**：设备执行 START STOP UNIT 命令时，若返回 Unit Attention（设备重启或复位），SCSI 核心层未重试导致操作失败。
6. **资源泄漏风险**：debugfs 初始化失败时的注销顺序错误可能导致资源泄漏或空指针访问；FW 信息异常时可能导致驱动初始化异常。

#主要错误日志
- `invalid phy number from FW`
- `invalid queue count from FW`
- `iopoll queue count %d cannot exceed or equal 16, using default 0`

#问题根因
1. **libsas BUG()**：函数 `sas_add_parent_port` 在处理添加 PHY 到端口时，直接调用 `BUG_ON` 进行错误处理，且未对零地址 PHY 的场景进行有效兼容，导致内核崩溃。
2. **控制器复位**：逻辑仅检查 `phy_state`，遗漏了对 `phy->enabled` 状态的检查与使能操作。
3. **PHY 超时处理**：拓扑扫描中仅处理了 PHY 变化，未针对 "旧状态 Ready 且新状态 Not Ready" 的 PHY Reset 超时场景进行复位操作。
4. **软复位时序**：代码在发送 H2D FIS 后未插入规定的延时等待。
5. **SCSI 启停命令**：SCSI 核心层对 passthrough 命令缺乏针对 Unit Attention 的自动重试机制。
6. **debugfs 安全性**：debugfs 文件创建与 trigger_dump 节点的创建顺序不合理；显示函数未检查数据是否已生成。

#解决方案
1. **修复 libsas 崩溃**：将 `sas_add_parent_port` 移至 `sas_expander.c` 并重构为 `sas_ex_add_parent_port`，引入 `sas_port_add_ex_phy` 辅助函数正确处理端口添加逻辑，移除 `BUG_ON` 调用，避免内核崩溃。
2. **修复 PHY 使能逻辑**：在控制器复位流程中，增加对 `sas_phy->phy->enabled` 的判断，对所有未被用户禁用的 PHY 执行 `hisi_sas_phy_enable`。
3. **增加 PHY 复位**：在扫描拓扑时，若检测到 PHY 状态由 Ready 变为 Not Ready，调用 `hisi_sas_notify_phy_event` 触发 `HISI_PHYE_LINK_RESET` 进行复位。
4. **修复软复位时序**：在发送 H2D FIS 之前添加 `usleep_range(900, 1000)` 延时。
5. **增加命令重试机制**：在 `scsi_execute_cmd` 中增加 `scsi_check_passthrough` 逻辑，允许 SCSI 中间层对特定错误（如 Unit Attention 0x29）进行重试；更新 `sd_start_stop_device` 使用该机制。
6. **优化 debugfs 与初始化**：调整 debugfs 注册与注销优先级；推迟 `trigger_dump` 创建时机至初始化末尾；增加 `debugfs_dump_is_generated_v3_hw` 检查；增加 `fw_info_check` 校验物理层和队列数合法性；开启默认中断聚合。
