#标题
usbnet: 在可抢占代码中使用 smp_processor_id() 触发内核警告

#内核版本
OLK-6.6

#问题现象
系统运行时触发内核警告 "BUG: using smp_processor_id() in preemptible code"，特别是在执行修改 MTU 操作时（如 dhcpcd 进程调用 `usbnet_change_mtu`）。调用栈显示警告源自 `usbnet_skb_return` 函数，表明在非原子上下文中调用了不应出现的 CPU ID 检查函数。

#主要错误日志
BUG: using smp_processor_id() in preemptible [00000000] code: dhcpcd/2879
caller is usbnet_skb_return+0x74/0x490 drivers/net/usb/usbnet.c:331

#问题根因
在 `usbnet_resume_rx` 函数中，调用链会执行到 `usbnet_skb_return`，该函数内部调用了 `netif_rx`。出于历史和可移植性原因，`netif_rx()` 通常在 softirq（软中断）或中断上下文中运行。当在进程上下文（如修改 MTU 导致的调用链）中直接调用时，由于未禁用软中断底半部，导致 `netif_rx` 内部错误地调用了 `smp_processor_id()`，从而触发了抢占检查警告。

#解决方案
在 `usbnet_resume_rx` 函数中增加 `local_bh_disable()` 和 `local_bh_enable()` 保护。这确保了在处理接收队列恢复和调用 `netif_rx` 期间，软中断底半部被禁用，从而满足 `netif_rx` 的上下文要求，消除了抢占上下文中的警告。
