#标题
ub: udma: 修复打印位置及链表遍历导致的潜在内核故障

#内核版本
OLK-6.6

#问题现象
1. 在 UDMA 驱动加载、卸载或资源分配/释放过程中，可能触发内核调度器警告或 Kernel Panic（如 scheduling while atomic）。
2. 在并发场景下销毁 UDMA 上下文或 Hugepage 资源时，可能导致内核崩溃或内存泄漏。
3. 进程上下文访问某些数据结构时可能发生 Use-after-free。

#主要错误日志
- "failed to alloc id, ret = %d.\n"
- "user specify id %u been used.\n"
- "all bitmaps have been used, bitmap_cnt = %u.\n"
- "no completely bitmap for Jetty group.\n"
- "mmap failed, j_id: %llu not exist\n"
- "mmap failed, j_id: %llu, uctx invalid\n"
- "async event for bogus queue number = %u.\n"
- "async event for bogus jfr number = %u.\n"
- "async event for bogus jfc number = %u.\n"
- "async event for bogus jetty group number = %u.\n"
- "num exceeds the maximum value.\n"
- "alloc jfr id range (%u - %u) failed, ret = %d.\n"

#问题根因
1. **死锁/原子上下文睡眠风险**：多个关键错误处理路径（如 `udma_idalloc_auto_grow`、`udma_specify_adv_id` 等）在持有自旋锁或 XArray 锁的情况下调用 `dev_err` 等打印函数。控制台打印可能发生睡眠，导致在原子上下文中调用睡眠函数，引发内核崩溃或调度异常。
2. **链表遍历释放缺陷**：在 `udma_destroy_hugepage` 和 `udma_free_ucontext` 函数中，使用 `list_for_each_entry` 遍历链表，并在循环体内删除当前节点，导致链表遍历破坏，引发内存访问错误或资源释放不完全。

#解决方案
1. **调整打印位置**：将所有在持有自旋锁、互斥锁或 XArray 锁时调用的 `dev_err`、`dev_warn` 等打印语句，移动到锁释放之后执行，避免在原子上下文中进行可能睡眠的操作。
2. **修复链表删除逻辑**：将链表遍历宏从 `list_for_each_entry` 修改为 `list_for_each_entry_safe`，并在循环体内显式调用 `list_del` 删除节点，确保在遍历过程中安全删除链表项。
3. **移除冗余打印**：删除 `udma_ctrlq_tp.c` 中 `udma_find_ue_idx_by_tpn` 函数里的冗余警告打印。
