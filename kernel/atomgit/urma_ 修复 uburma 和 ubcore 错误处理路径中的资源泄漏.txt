#标题
urma: 修复 uburma 和 ubcore 错误处理路径中的资源泄漏

#内核版本
OLK-6.6

#问题现象
内核在处理 URMA 相关的 ioctl 命令和 mmap 操作时，若遇到参数校验失败或内存分配失败等异常情况，会导致内存泄漏或返回值错误，具体表现为：
1. 在调用 `ubcore_get_topo_info` 时，若从用户空间拷贝数据失败，之前分配的内存 `arg` 未被释放。
2. 在调用 `uburma_cmd_free_token_id` 时，若 token ID 校验不匹配，引用计数未正确释放（uobj 未 put）。
3. 在调用 `uburma_mmap` 时，若底层 mmap 操作失败或后续内存分配失败，未返回正确的错误码或未执行清理逻辑。
4. `ubcore_get_uvs_init_res_dump` 函数中存在返回未初始化变量的问题。

#主要错误日志
未知

#问题根因
代码在多个错误处理路径（error path）中存在逻辑缺陷：
1. `ubcore_genl_admin.c` 中 `ubcore_get_topo_info` 函数在 `copy_from_user` 失败时直接返回，遗漏了 `kfree(arg)` 操作。
2. `uburma_cmd.c` 中 `uburma_cmd_free_token_id` 函数在 token ID 不匹配时直接返回，遗漏了 `uobj_put_del(uobj)` 操作，导致对象引用计数泄漏。
3. `uburma_dev_ops.c` 中 `uburma_mmap` 函数在 `ops->mmap` 失败时未跳转到清理逻辑，且 `kzalloc` 失败时未设置返回值 `ret` 为 `-ENOMEM`。
4. `ubcore_netlink.c` 中 `ubcore_get_uvs_init_res_dump` 函数未初始化局部变量 `ret`，可能返回未定义的值。

#解决方案
1. 在 `ubcore_get_topo_info` 的 `copy_from_user` 失败分支中添加 `kfree(arg)` 释放内存。
2. 在 `uburma_cmd_free_token_id` 的 token ID 校验失败分支中添加 `uobj_put_del(uobj)` 释放对象引用。
3. 修正 `uburma_mmap` 的控制流：
   - 若 `ops->mmap` 失败，跳转到 `out` 标签退出。
   - 若 `kzalloc` 分配失败，将 `ret` 设置为 `-ENOMEM`。
4. 在 `ubcore_get_uvs_init_res_dump` 中将变量 `ret` 初始化为 0。
