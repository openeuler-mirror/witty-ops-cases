#标题
f2fs: inode销毁时因页缓存泄漏触发kernel BUG

#内核版本
OLK-6.6

#问题现象
在挂载损坏的f2fs镜像时，内核触发kernel BUG并打印调用栈。
具体表现为：
1. f2fs文件系统校验失败，报错“corrupted inline inode”及“Wrong SSA boundary”。
2. 随后触发kernel BUG at fs/inode.c:753，导致系统异常。
3. 调用栈显示崩溃发生在clear_inode函数中，由f2fs_evict_inode调用链触发。

#主要错误日志
loop0: detected capacity change from 0 to 40427
F2FS-fs (loop0): Wrong SSA boundary, start(3584) end(4096) blocks(3072)
F2FS-fs (loop0): Can't find valid F2FS filesystem in 1th superblock
F2FS-fs (loop0): invalid crc value
F2FS-fs (loop0): f2fs_convert_inline_folio: corrupted inline inode ino=3, i_addr[0]:0x1601, run fsck to fix.
------------[ cut here ]------------
kernel BUG at fs/inode.c:753!
RIP: 0010:clear_inode+0x169/0x190 fs/inode.c:753

#问题根因
在f2fs_truncate()函数的错误处理路径中存在页缓存泄漏。
当f2fs_convert_inline_inode()内部调用f2fs_grab_cache_folio()创建了第0页（page #0）后，若后续的完整性检查失败导致函数返回错误（如-EFSCORRUPTED），代码直接返回而未释放该页缓存。随后在inode销毁流程（clear_inode）中，内核检测到inode->i_data.nrpages不为零，从而触发kernel BUG。

#解决方案
在f2fs_truncate()函数中，当f2fs_convert_inline_inode()返回错误时，增加对第0页的截断操作（truncate_inode_pages_range），强制释放已分配的页缓存，防止在inode销毁时发生内存泄漏。
