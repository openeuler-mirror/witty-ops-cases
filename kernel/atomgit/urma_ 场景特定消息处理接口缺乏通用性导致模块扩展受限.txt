#标题
urma: 场景特定消息处理接口缺乏通用性导致模块扩展受限

#内核版本
OLK-6.6

#问题现象
ubcore 模块当前为聚合设备实现并暴露的场景特定消息处理接口（如 bonding user msg 接口）缺乏通用性。这导致外部模块无法方便地扩展自定义的消息处理能力，破坏了 ubcore 的模块完整性，限制了系统的扩展性。

#主要错误日志
未知

#问题根因
Ubcore 当前实现的场景特定消息处理接口是为特定聚合设备场景硬编码的，没有提供通用的注册与分发机制。这种设计导致外部模块无法注册自定义的消息处理方法，只能依赖 ubcore 内部有限的预设逻辑，从而导致扩展困难，破坏了模块化设计。

#解决方案
在 ubcore 中实现并暴露一套通用的消息处理接口。
1. 引入 `ubcore_comm_msg` 结构体替代原有的 `ubcore_net_msg`，增加 `protocol_id` 字段用于协议区分。
2. 新增通用的注册/注销接口 `ubcore_register_comm_msg_handler` 和 `ubcore_unregister_comm_msg_handler`，外部模块可通过指定 `protocol_id` 注册自定义处理函数。
3. 重构消息分发逻辑，根据 `protocol_id` 将消息路由到已注册的处理函数中；`protocol_id` 为 0 保留给 ubcore 内部使用，其他 ID 由外部模块扩展使用。
4. 删除原有的专用接口（如 `ubcore_net_send_bonding_user_msg`），将相关功能迁移至新的通用接口框架下。
