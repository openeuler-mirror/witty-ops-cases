#标题
ath11k: GTK重密钥期间组播数据包丢包问题

#内核版本
OLK-6.6

#问题现象
在GTK（Group Temporal Key）重密钥期间，组播/组地址数据包发生丢包。
具体表现为：在STA/AP/MESH模式下，当进行密钥更新时，mac80211连续下发清除旧密钥和安装新密钥的指令，导致固件与硬件状态不一致，进而引发组播数据包传输失败。该问题在高负载或时间敏感场景下尤为明显。只有重新设置相同的密钥才能恢复固件状态并允许数据包传输。

#主要错误日志
- `ath11k_warn(ab, "failed to set group key of idx %d for vdev %d: %d\n", i, arvif->vdev_id, ret);`
- `ath11k_warn(ab, "ath11k_install_key failed (%d)\n", ret);`

#问题根因
在GTK重密钥过程中，mac80211会在同一上下文中先清除旧密钥（如果存在），随后安装新密钥。这导致ath11k驱动连续发送两个WMI命令：清除旧密钥和在同一槽位安装新密钥。
在高负载或特定时序条件下，固件可能异步处理这两个命令，导致固件认为密钥已被清除，而硬件实际上仍持有有效密钥。这种硬件与固件之间的不一致状态导致了组播数据包被丢弃。
该问题在引入提交 `436a4e886598 ("ath11k: clear the keys properly via DISABLE_KEY")` 后暴露出来，因为该提交使得主机能够显式清除组密钥，从而触发了上述竞态条件。

#解决方案
修改驱动逻辑，限制组密钥的清除操作，避免在重密钥期间出现竞态条件：
1. **限制组密钥清除**：在所有模式（AP, STA, MESH）下，禁止在有关联站点的情况下清除组密钥。重密钥时，直接在原密钥之上设置新密钥，避免“清除-设置”的连续操作。
2. **AP模式特例处理**：仅在AP模式下且无关联站点时允许清除组密钥，以支持从安全模式（如WPA2/WPA3）向开放模式切换。
3. **引入计数器与标志位**：
   - 增加 `num_stations` 计数器，用于跟踪BSS下的站点数量。
   - 增加 `reinstall_group_keys` 标志位，用于标记密钥重安装状态。
4. **恢复机制**：在AP和Mesh模式下，当第一个站点关联时，如果 `reinstall_group_keys` 标志被置位，则重新安装最后一次设置的组密钥。这确保固件能够从之前无站点期间可能发生的密钥清除竞态中恢复。
