#标题
mse102x 驱动 RX 错误处理导致数据包堆积或通信异常

#内核版本
OLK-6.6

#问题现象
当 MSE102x 设备在 SPI 通信过程中 CMD_RTS 命令受到干扰损坏时，驱动程序未能正确处理接收流程。具体表现为：驱动检测到无效的 RTS 响应后直接返回，未将 SPI FIFO 中已存在的以太网帧数据移出。这导致后续通信无法正常进行，可能造成数据包堆积或设备通信死锁。

#主要错误日志
- `Unexpected response (0x%04x)` （推断日志格式，对应 `invalid_rts` 计数增加）
- `Invalid frame length: %d` （推断日志格式，对应 `invalid_len` 计数增加）
- `Drop frame` （推断日志格式，对应修复后新增的调试打印）

#问题根因
在 SPI 通信中，如果 CMD_RTS 命令被干扰损坏，MSE102x 设备不允许重传该命令。此时，驱动程序原本的处理逻辑是直接返回退出，但这会导致 SPI FIFO 中残留的以太网帧数据未被移出。由于设备逻辑认为帧数据已被发送，但主机未读取，导致 FIFO 状态不一致，阻碍了后续的正常通信。

#解决方案
修改接收路径的错误处理逻辑。当检测到无效的 CMD_RTS 响应或无效帧长度时，不再直接返回，而是设置 `drop` 标志位。在 `drop` 标志置位的情况下，强制假设帧长度为最大可能值（VLAN_ETH_FRAME_LEN），继续执行 SPI 读取操作以将 FIFO 中的数据“排空”或丢弃。这确保了即使命令受损，SPI FIFO 中的数据也能被正确移出，从而恢复设备的正常通信状态。
