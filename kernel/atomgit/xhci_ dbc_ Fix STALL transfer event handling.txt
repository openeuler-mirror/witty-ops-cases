#标题
xhci: dbc: Fix STALL transfer event handling

#内核版本
OLK-6.6

#问题现象
xHC DbC（Debug Capability）在处理批量传输时可能触发 STALL 错误事件，导致驱动错误地刷新所有待处理请求，造成数据传输中断或数据丢失。此外，在特定情况下（如主机向非停止的端点发送 ClearFeature(ENDPOINT_HALT) 请求时），xHC DbC 可能产生虚假的 STALL 事件，导致驱动程序处理逻辑异常。

#主要错误日志
- "DbC Endpoint halted"
- "DbC Endpoint halt cleared"
- "Stall error at bulk TRB %llx, remaining %zu, ep deq %llx"
- "Give back stale stalled req"
- "no matched request"
- "tx error %d detected"

#问题根因
1. **错误的 STALL 处理逻辑**：之前的实现在检测到端点停止时，会将 DbC 状态设置为 DS_STALLED 并刷新该端点的所有待处理请求。这是不正确的，因为 DbC 规范中没有“Stalled”状态，且端点停止是针对每个端点的状态，不应影响全局 DbC 状态。
2. **虚假 STALL 事件处理缺失**：硬件可能在未进行实际传输时产生 STALL 事件（例如传输长度为 0 且端点已停止在当前 TRB）。旧代码未识别这种“虚假”事件，错误地返还了请求，导致硬件后续重启传输时 TRB 与请求不同步。
3. **重传风险**：在真实的 STALL 传输错误中，硬件在收到 ClearFeature(ENDPOINT_HALT) 后会从停止的 TRB 重新开始处理。如果驱动已将该 TRB 对应的请求返还，硬件可能重用该 TRB 导致重传或数据损坏。

#解决方案
1. **移除全局 DS_STALLED 状态**：删除了驱动自定义的 DS_STALLED 状态，因为 DbC 状态机中只存在 Configured 等状态，停止状态应通过端点级别的标志位跟踪。
2. **引入端点级别停止标志**：在 `dbc_ep` 结构体中新增 `halted` 位，用于精确跟踪每个端点的停止状态，替代原有的全局状态判断。
3. **优化 STALL 事件处理逻辑**：
   - 不再刷新所有请求。
   - 如果 STALL 事件显示传输长度为 0 且端点停止在当前 TRB 上，判定为虚假 STALL 事件，保留请求不返还，防止后续传输不同步。
   - 如果是真实传输错误导致的 STALL，将对应的 TRB 转换为 No-op（空操作）后再返还请求。这确保硬件重启时不会重复处理该 TRB，而是跳过它继续执行下一个。
4. **修正端点停止后的恢复流程**：通过 `handle_ep_halt_changes` 函数处理端点停止状态的变迁，在端点停止清除时正确触发铃声以恢复传输。
