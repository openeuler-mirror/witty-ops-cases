#标题
alloc_fdtable 调用约定重构及错误处理修复

#内核版本
OLK-6.6

#问题现象
内核在调用 `alloc_fdtable` 分配文件描述符表时，若遇到内存不足或文件描述符限制问题，调用者处理逻辑复杂且存在潜在的处理不一致风险。具体表现为：
1.  `expand_fdtable()` 和 `dup_fd()` 在分配失败时需要检查返回值是否为 NULL 以及分配的表大小是否满足需求，逻辑冗余且容易出错。
2.  在 32 位系统上，当请求的文件描述符数量小于 256 时，计算出的容量可能不符合预期（原逻辑可能产生 64 或 128 的小容量），导致后续操作异常。
3.  若 `sysctl_nr_open` 在检查与分配之间被修改为更小的值，原代码在 `expand_fdtable()` 中可能释放新分配的表并返回 -EMFILE，但未统一在分配函数中处理该边界情况。

#主要错误日志
未知

#问题根因
1.  **调用约定不清晰**：`alloc_fdtable` 原本接收的参数含义模糊（基于 1024/sizeof(ptr) 计算后的值），导致调用者（如 `dup_fd`）需要进行复杂的反向计算。
2.  **错误处理不统一**：原 `alloc_fdtable` 在失败时返回 NULL，且不检查 `sysctl_nr_open` 缩减导致的容量不足问题，迫使调用者额外增加检查逻辑。
3.  **容量计算逻辑不合理**：原逻辑在 32 位系统上允许分配 64 或 128 个 slot 的小容量 fdtable，而实际上对于 32 位系统，期望的最小容量应为 256（即 1KB fd array），以保持内存分配的合理性。

#解决方案
1.  **重构调用约定**：修改 `alloc_fdtable` 函数，直接接收期望的 slot 数量 (`slots_wanted`) 作为参数，简化调用者逻辑。
2.  **统一错误返回**：修改 `alloc_fdtable` 使其在失败时返回 `ERR_PTR()`（包含具体的错误码如 -ENOMEM 或 -EMFILE），而不是返回 NULL。
3.  **优化容量计算**：在 `alloc_fdtable` 内部重新设计计算逻辑：
    *   对于 32 位系统，强制最小 slot 数量为 256。
    *   使用 `roundup_pow_of_two` 对齐 slot 数量。
    *   在函数内部处理 `sysctl_nr_open` 限制，若计算出的容量小于请求数量直接返回 -EMFILE。
4.  **简化调用者**：更新 `expand_fdtable` 和 `dup_fd`，使其直接传递期望的 slot 数量，并使用 `IS_ERR` 检查返回值，移除冗余的二次检查代码。
