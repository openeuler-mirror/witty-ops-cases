#标题
riscv: mm: vmemmap 地址越界访问缺陷

#内核版本
OLK-6.6

#问题现象
在 RISC-V 架构使用 sparse vmemmap 内存模型时，如果 `phys_ram_base` 所在内存节的起始页帧号（pfn）小于 `phys_ram_base` 对应的页帧号，内核在初始化该内存节中不可用范围的 struct page（调用 `init_unavailable_range`）时，计算出的 vmemmap 虚拟地址会低于 `VMEMMAP_START`。这导致 vmemmap 地址越界，侵入到了 `PCI_IO_END` 以下的地址空间，可能引发内存映射错误或系统异常。

#主要错误日志
未知

#问题根因
在 sparse vmemmap 模型中，vmemmap 虚拟地址的计算公式为 `((struct page *)VMEMMAP_START - (phys_ram_base >> PAGE_SHIFT))`。
根本原因在于：内核在初始化 struct page 时，实际上是从 `phys_ram_base` 所属的同一个 memory section 的起始位置开始的。
举例：若 `phys_ram_base` 为 0x82000000（pfn 0x82000），其所属 section 的起始 pfn 实际为 0x80000。在计算 pfn 0x80000 对应的 struct page 虚拟地址时，公式变为 `VMEMMAP_START - 0x2000`，导致计算结果低于 `VMEMMAP_START`，造成地址越界。

#解决方案
引入一个新的变量 `vmemmap_start_pfn`，将其初始化为与 memory section 大小对齐的起始 pfn（通过 `round_down(phys_ram_base, VMEMMAP_ADDR_ALIGN) >> PAGE_SHIFT` 计算得出）。
修改 vmemmap 宏定义，使用 `vmemmap_start_pfn` 替代原有的 `(phys_ram_base >> PAGE_SHIFT)` 进行地址计算。这确保了计算 vmemmap 基地址时使用的是 section 对齐的起始 pfn，从而保证所有 struct page 的计算地址均位于 `VMEMMAP_START` 之上的合法区域。
