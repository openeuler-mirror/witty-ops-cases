#标题
sw64 MSI 中断迁移时 CPU 热插拔导致中断长期失效问题

#内核版本
OLK-6.6

#问题现象
在频繁进行 CPU 热插拔操作的场景下，当受管理的中断其亲和性掩码中最后一个活跃 CPU 下线时，中断会持续处于禁用状态，直到原亲和性掩码中有其他 CPU 上线才能恢复。随着热插拔操作频繁进行，亲和性掩码可能被逐渐收窄，导致可用于恢复中断的 CPU 池减少，系统响应能力下降。

#主要错误日志
未知

#问题根因
在 CPU 热插拔导致中断迁移的过程中，代码逻辑存在缺陷：
1. 中断亲和性掩码被不当收窄，导致后续可用于处理中断的候选 CPU 池变小。
2. 缺少对 `irq_data_update_effective_affinity` 的正确更新，导致中断必须等待特定 CPU 上线才能恢复，而非原亲和性掩码中的任意 CPU。
3. 对于受管理的中断，在未启动状态下未正确处理亲和性设置逻辑，影响了中断迁移的健壮性。

#解决方案
1. 在 Kconfig 中启用 `GENERIC_IRQ_EFFECTIVE_AFF_MASK` 配置，支持有效亲和性掩码机制。
2. 修改 `sw64_set_affinity` 逻辑：
   - 放宽中断启动状态的检查条件，允许未启动但受管理的中断进行亲和性设置。
   - 调整 `move_in_progress` 和目标 CPU 在线状态的检查顺序，仅在目标 CPU 在线时才进行忙碌检查，避免死锁或逻辑错误。
   - 在设置亲和性完成时，调用 `irq_data_update_effective_affinity` 更新为实际目标 CPU，而非搜索掩码，确保内核记录正确的有效亲和性。
3. 在 `__assign_irq_vector` 分配向量时，同步调用 `irq_data_update_effective_affinity` 更新有效亲和性，保留原始亲和性掩码以维持较大的候选 CPU 池，确保只要原掩码内有任意 CPU 上线，中断即可恢复。
