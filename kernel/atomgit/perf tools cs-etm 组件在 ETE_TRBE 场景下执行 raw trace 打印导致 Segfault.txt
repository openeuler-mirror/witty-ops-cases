#标题
perf tools cs-etm 组件在 ETE/TRBE 场景下执行 raw trace 打印导致 Segfault

#内核版本
OLK-6.6

#问题现象
在使用 CORESIGHT=1 和 CSTRACE_RAW=1 编译选项构建 perf 工具后，执行 `perf report --dump` 命令对 ETE（Embedded Trace Extension）和 TRBE（Trace Buffer Extension）的跟踪数据进行解码时，程序异常终止，触发段错误。该问题不仅导致调试信息打印失败，还影响了对 trace 质量、损坏情况或解码问题的调查功能。

#主要错误日志
未知

#问题根因
1. **硬件特性不匹配**：原始代码尝试打印 CoreSight 格式化的 trace frames，但这些 frames 在 ETE + TRBE 跟踪场景下并不存在，属于不必要的调用。
2. **库兼容性缺陷**：在使用版本较旧（<= 1.8.0）的 OpenCSD 库时，上述不必要的打印调用会触发库内部的额外错误，直接导致 perf 工具发生 Segfault。
3. **代码逻辑冗余**：原始实现中的辅助函数 `cs_etm_decoder__init_raw_frame_logging` 重复执行了部分初始化调用，而这些初始化在主流程中已经完成。

#解决方案
修改 `cs-etm-decoder.c` 文件，移除独立的 `cs_etm_decoder__init_raw_frame_logging` 辅助函数，将其核心逻辑合并至 `cs_etm_decoder__init_def_logger_printing` 函数中，并增加条件判断：
1. **增加硬件格式判断**：仅在 `d_params->formatted` 为真（即硬件实际生成了格式化的 CoreSight trace frames）时，才设置 raw frame 打印机。
2. **保留操作类型判断**：继续检查操作类型是否为 `CS_ETM_OPERATION_PRINT`（对应 `--dump` 参数）。
3. **删除冗余初始化**：移除原代码中重复的 `ocsd_def_errlog_init` 等初始化调用，消除冗余逻辑并规避旧版本 OpenCSD 库的缺陷。
