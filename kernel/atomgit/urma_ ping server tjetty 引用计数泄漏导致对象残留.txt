#标题
urma: ping server tjetty 引用计数泄漏导致对象残留

#内核版本
OLK-6.6

#问题现象
在 ping server 处理流程中，`tjetty` 对象的引用计数未被正确释放，导致对象在哈希表中持久存在无法释放，造成内存泄漏及资源残留。

#主要错误日志
未知

#问题根因
引用计数管理逻辑存在缺陷：
1. 在发送流程 `ping_wq_on_recved` 中，`tjetty` entry 的引用计数被增加，但在发送完成回调 `ping_wq_on_sended` 中未能正确对应释放。
2. `ping_tjetty_put` 接口设计存在逻辑漏洞，其通过 `dst_eid` 在哈希表中查找对象并尝试减少引用计数，但在并发或特定流程下查找失败时直接返回，导致引用计数无法释放。
3. `__ping_tjetty_add` 函数中不论是新创建 entry 还是找到已存在 entry，都错误地调用了 `kref_get`，导致引用计数增加混乱。

#解决方案
1. 重构引用计数传递机制，引入 `ubmgr_ping_resp_ctx` 结构体，将 `tjetty` entry 指针与发送上下文绑定，直接通过指针传递引用关系，移除不可靠的查找逻辑。
2. 修正 `ping_tjetty_put` 实现，接收 entry 指针作为参数，直接对其执行 `kref_put` 操作，若引用计数归零则立即从哈希表删除并释放内存。
3. 修正 `__ping_tjetty_add` 逻辑，仅在查找到已存在 entry 时增加引用计数，新创建的 entry 不再额外增加引用计数。
4. 调整发送与完成回调的引用计数配对逻辑，确保发送请求期间持有引用，发送完成后（无论成功与否）正确释放引用。
