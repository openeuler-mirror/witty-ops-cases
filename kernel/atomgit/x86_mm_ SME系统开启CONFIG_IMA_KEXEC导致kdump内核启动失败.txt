#标题
x86/mm: SME系统开启CONFIG_IMA_KEXEC导致kdump内核启动失败

#内核版本
OLK-6.6

#问题现象
在开启 SME（Secure Memory Encryption）功能的系统中，若内核编译开启了 CONFIG_IMA_KEXEC 选项，当系统崩溃进入 kdump 内核时，会发生通用保护故障并导致内核恐慌。系统打印 IMA 相关的初始化信息后触发 Oops，最终重启。

```
Loading compiled-in module X.509 certificates

ima: Allocated hash algorithm: sha256
Oops: general protection fault, probably for non-canonical address 0xcfacfdfe6660003e: 0000 [#1] PREEMPT SMP NOPTI
CPU: 0 UID: 0 PID: 1 Comm: swapper/0 Not tainted 6.11.0-rc2+ #14
...

 <TASK>

```
（调试模式下可见：ima: ima_load_kexec_buffer, buffer:0xcfacfdfe6660003e, size:0xe48066052d5df359）

#问题根因
在 `early_memremap_is_setup_data()` 和 `memremap_is_setup_data()` 函数中，判断给定物理地址是否属于 setup_data 区域的逻辑存在错误。代码中的 `len` 变量来源于 `struct setup_data` 的 `len` 字段，该字段仅表示数据部分的长度，不包含结构体自身的长度。原代码在计算范围结束地址时仅使用了 `paddr + len`，遗漏了结构体头部的大小，导致范围计算过小。这使得 SME 系统在处理 IMA kexec buffer 时错误的将其排除在 setup_data 范围之外，进而导致该内存区域未被正确解密，访问未解密的地址引发通用保护故障。

#解决方案
修正 `arch/x86/mm/ioremap.c` 中的内存范围计算逻辑。在 `memremap_is_setup_data()` 和 `early_memremap_is_setup_data()` 函数中，计算结束地址时加上 `sizeof(struct setup_data)`，即将判断条件修正为 `phys_addr < (paddr + sizeof(struct setup_data) + len)`，确保整个 setup_data 结构体（头部+数据）都在检测范围内，从而在 SME 系统上能正确识别并对该区域进行解密处理。
