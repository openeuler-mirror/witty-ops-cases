#标题
网络流控调度器多个类qlen_notify接口非幂等导致内核崩溃

#内核版本
OLK-6.6

#问题现象
在特定网络流控场景下（如 HTB 嵌套使用 codel/qfq 等调度器，或进行配置变更操作时），由于 `qlen_notify` 等接口实现非幂等，当队列长度在数据包丢弃过程中降为 0 时触发内核崩溃，或因重复删除链表节点导致系统异常。

#主要错误日志
未知

#问题根因
内核网络调度器子系统中的多个 qlen_notify 回调函数（如 sch_drr, sch_ets, sch_qfq, sch_hfsc, sch_htb）实现非幂等。
具体表现为：
1. sch_codel 和 sch_fq_codel 中存在错误的防御性检查，当 `sch->q.qlen` 为 0 时不调用 `qdisc_tree_reduce_backlog`，这会导致上层 HTB 调度器状态更新滞后，进而在后续操作中引发崩溃。
2. 各底层调度器的 `qlen_notify` 或相关链表操作函数（如 `list_del`、`rb_erase`、`htb_deactivate`）缺乏对重复调用或空状态的处理。当数据包被丢弃且触发 backlog 更新时，可能导致链表节点被重复删除（double remove）或操作已处于非活跃状态的类对象，从而触发 kernel panic 或 oops。

#解决方案
修补调度器代码，使相关操作接口具备幂等性：
1. 移除 sch_codel 和 sch_fq_codel 中对 `sch->q.qlen` 的前置检查，允许在 qlen 为 0 时正常更新 backlog。
2. 将各调度器中的 `list_del` 调用改为 `list_del_init`，防止重复删除链表节点。
3. 在 sch_hfsc 和 sch_qfq 的相关函数中增加节点状态或链表空检查（如 `RB_EMPTY_NODE`、`list_empty`），确保仅当节点处于活跃/有效状态时才执行移除操作。
4. 将 sch_htb 中的 `htb_deactivate` 改为幂等实现，增加 `prio_activity` 检查，避免重复注销。
