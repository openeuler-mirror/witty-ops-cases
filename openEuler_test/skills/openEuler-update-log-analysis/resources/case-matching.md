# 案例库匹配规则

> 本文件定义案例库路径、检索方式与三级拦截判定规则。分析每个失败用例时按此执行。

## 案例库路径

```
/root/xys/gitcode/witty-ops-cases/openEuler_test/update_test/
```

按模块分子目录：

- `update_test/pkgmanage/`
- `update_test/kernel/`
- `update_test/pkgserver/`
- `update_test/docker/`
- `update_test/pkgcmd/`

案例文件为 `.md` 或 `.txt`，每条案例包含字段：测试用例、系统版本、**是否问题**（否/是/待确认）、关联issue、错误日志、根因、影响范围、解决方案、关键词。

> 跨模块同根因的用例可匹配其他模块的案例（例如 pkgserver 的 httpd 失败可匹配 pkgcmd 的 httpd 案例）。

## 检索方式

1. **优先用 `rag_search` 工具**（若可用）在相关知识库中检索类似故障，调整 `keyword_weight` 并排除不相关 chunk 后重试。
2. 若无 rag_search，则用 `grep`/`rg` 在案例库目录下按关键词检索：
   - 先用用例名 `oe_test_<name>` 或 LTP 用例名精确匹配。
   - 再用报错日志关键短语（如 `Connection refused`、`Unit not found`、`scriptlet failed`）模糊匹配。
3. 每次检索都要在报告"检索记录"中记录用过的关键词与命中结果（命中案例文件名 或 "无匹配"）。
4. 若第一次检索结果不理想，换 2-3 组关键词重试，并记录每次结果。

## 三级拦截判定

每个失败用例经检索后落入以下之一：

### ✅ 自动拦截

案例库命中且案例"是否问题"字段明确（"否" 或 "是"）：

- **非问题(环境/预期行为)** —— 案例标注"是否问题: 否"。环境限制、版本不支持、预期行为差异等。
- **已知问题(案例库命中)** —— 案例标注"是否问题: 是"。已知缺陷或配置问题，已有案例记录根因与解决方案。
- **已有关联分析(连锁同根因)** —— 与本报告或他模块报告中已拦截用例同根因（如 httpd 连锁导致 httpd-init 失败），可直接引用，无需重复分析。

### ⚠️ 待确认

案例库部分匹配，但案例标注"是否问题: 待确认"，或案例描述与当前现象相似但未最终定性。

- 仍需在报告中列出：部分匹配案例、检索关键词、AI 初步根因、不确定点、关键日志证据、需人工确认的问题。
- 待人工将案例定性为"否/是"后，下次可升级为 ✅ 自动拦截。

### ❌ 未拦截

案例库无任何匹配。**必须完整分析**，按 `report-template.md` 中 ❌ 段落给出 6 字段 + 案例草稿：

- 测试执行上下文
- 关键日志证据（关键日志前后 5 行）
- 分析逻辑链（逐步推演）
- 检索记录
- 根因推断（标注是否与包更新质量相关）
- 修复建议
- 📥 案例草稿（blockquote，格式同案例库 `.md`，便于人工审核后直接入库）

## 判定铁律

- 只有案例库中**明确写着"非问题"**的才算非问题，**不许自行定义是否为问题**。
- 案例库未找到明确表示非问题的报错**一律视为问题**，归入 ❌ 未拦截。
- 案例"是否问题"字段缺失时按"待确认"处理，归入 ⚠️ 待确认。

## 案例草稿格式（嵌入 ❌ 段落）

用 blockquote `>` 包裹，与案例库 `.md` 字段一致：

```
> # <现象标题>
>
> ## 现象
> <一句话现象>
>
> ## 基本信息
> - 测试用例: oe_test_...
> - 系统版本: openEuler-<version>
> - 是否问题: 否 / 是 / 待定
> - 关联issue: 无
>
> ## 错误日志
> ```
> <关键日志>
> ```
>
> ## 根因
> <根因说明>
>
> ## 影响范围
> <受影响用例/模块>
>
> ## 解决方案
> <修复建议>
>
> ## 关键词
> <逗号分隔关键词>
```

## 拦截率计算

拦截率 = (✅ 自动拦截 + ⚠️ 待确认) / 总失败用例（排除后）。
周报中可单独统计 ✅ 拦截率与含待确认的口径，并在跨周对比时统一口径。
