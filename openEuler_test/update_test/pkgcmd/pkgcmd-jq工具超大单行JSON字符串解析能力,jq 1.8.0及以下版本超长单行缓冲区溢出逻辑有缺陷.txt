# 问题根因
1、 jq 1.8.0 及 1.8.0以下版本存在的缺陷：
    jq 1.8.0 的词法扫描器（lexer）有固定长度输入缓冲区，存在两个关键问题：
        a、超长单行缓冲区截断：单行文本超过内部缓冲阈值（4096/16384 字节）时，词法分析器会拆分多行读取，但行号计数器不会同步自增，超长字符串整行被错误识别为和上一行同行号。
        b、input_line_number 行号计数逻辑漏洞：仅在读取到 \n 换行符时才增加行号；超长单行被缓冲区切分多次读取，但中间无换行，行号保持不变。
    测试用例构造 4095、16380 字节超长字符串，刚好触发该漏洞，第二行行号仍输出1，和脚本写死的基准文件匹配，测试 “假通过”。

2、 1.8.1~1.8.2 修复的核心补丁（导致用例失败）
    从 1.8.1 开始官方修复输入缓冲区与行号统计逻辑，1.8.2 稳定落地：
        a、动态扩容输入缓冲区：不再使用固定大小 buffer，任意长度单行完整读取，不会截断拆分。
        b、行号计数逻辑标准化：严格按照物理换行分隔行，只要是独立一行，无论单行多长，input_line_number 都会输出正确递增行号；只要输入存在\n换行，行号 + 1，不受单行内容长度影响。
        c、同步修复多起安全漏洞（CVE 缓冲区截断、空字节解析缺陷），解析行为严格遵循 JSON 文本分行标准，不再存在超长行号错乱问题。

# 解决方案
1、随着jq软件包更新到1.8.2版本，input_line_number 行号计数逻辑漏洞被修复，需要更新测试用例基准/tmp/diff_result内容为：1 1 3 ——> 1 2 3

# 备注
1、jq 有一个特性：当输入行长度正好是 4095 的倍数 时，行号计算可能出错。
2、jq input_line_number 作用：打印当前处理行的行号，常用于定位 JSON 语法错误所在行数。
